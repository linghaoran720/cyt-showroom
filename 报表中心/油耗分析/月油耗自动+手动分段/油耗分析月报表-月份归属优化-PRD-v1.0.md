## 1. 概述

### 1.1 项目背景

财运通物流管理SaaS系统的油耗分析模块，采用"分段定油耗"模式，以"满到满"（两次加满油之间）作为基本计算单元，生成满到满报表。部分客户需要按月度管理油耗时，系统在满到满基础上按固定逻辑合并为月度报表：**上月最后一次加满 → 本月最后一次加满 = 本月油耗**。

**当前痛点**：客户会管理司机加油行为，正常情况下要求司机在每月月底/月初回公司加满油（公司采购油价便宜，且不加满可多装货）。但实际中部分车辆因运输任务无法在月底回来，只能月初回来补满，导致：

1. 一部分车月底加满：上月底→本月底，计算相对合理
2. 一部分车月初加满：本月初→下月初，系统会把两个月油耗合并到一个月
3. 一部分车跨月：上月初→下月初，本月完全无加满记录，系统显示无数据

### 1.2 项目目标

- 优化满到满周期节点的自动选取逻辑，适配"月底/月初加满"的不同业务节奏
- 减少因固定逻辑导致的空月和数据堆积问题
- 提供人工修正能力作为兜底，覆盖极端场景
- 保持满到满基本计算模式不变，确保向下兼容

### 1.3 项目范围

**包含（In Scope）**
- 周期节点自动选取逻辑优化
- 人工修正周期节点能力
- 修正后自动重新计算
- 非满油加油记录纳入周期计算
- 车辆首条加油记录默认视为满油
- 司机离职结算场景
- 历史数据刷新兼容新逻辑

**不包含（Out of Scope）**
- 分段明细的分段逻辑（空重驶、国道高速等影响因素不变）
- 标准百公里油耗配置不变
- 奖惩系数配置不变
- 满到满的基本计算模式不变

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 满到满周期 | 两次加满油之间的时间区间，是油耗计算的基本单元 |
| 周期节点 | 作为满到满周期切割点的加满记录，决定周期的开始和结束 |
| 分段明细 | 在满到满周期内，根据空重驶、国道高速等影响因素切分的更细粒度计算单元 |
| 标准升数 | 根据里程和标准百公里油耗计算得出的理论油耗量 |
| 实际升数 | 周期内所有加油记录的加油量总和 |
| 升数差额 | 标准升数 - 实际升数，正数为节油，负数为超耗 |
| 分段定油耗 | 系统油耗计算模式之一，根据影响因素对行驶过程分段并分别匹配油耗标准 |

### 3.2 功能详细说明

#### F001：周期节点自动选取逻辑优化

##### 设计思路 - 三层机制

| 层级 | 机制 | 说明 |
|------|------|------|
| 第一层 | 系统自动计算（优化后） | 覆盖大部分标准及常见场景，减少人工介入 |
| 第二层 | 人工修正 | 对自动计算不合理的情况，允许手动调整周期节点 |
| 第三层 | 业务约束前提 | 客户需管理司机加满行为，确保合理时机加满 |

##### 自动计算规则

计算N月油耗时，需确定"开始节点"和"结束节点"：

**找开始节点（按优先级）**：

| 优先级 | 条件 | 开始节点 |
|--------|------|----------|
| 1（最高） | N-1月有加满记录 | N-1月**最后一次**加满 |
| 2 | N-1月无加满，N月有加满 | N月**第一次**加满（同时作为N-1月油耗的结束节点） |
| 3 | N-1月无加满，N月也无加满 | N月无油耗数据 |

**找结束节点（按优先级）**：

| 优先级 | 条件 | 结束节点 |
|--------|------|----------|
| 1（最高） | N月有加满记录 | N月**最后一次**加满 |
| 2 | N月无加满，N+1月有加满 | N+1月**第一次**加满 |
| 3 | N月无加满，N+1月也无加满 | N月无油耗数据 |

**N月油耗 = 开始节点 → 结束节点**

##### 场景验证（计算2月油耗）

| 场景 | 1月加满 | 2月加满 | 3月加满 | 2月油耗周期 |
|------|---------|---------|---------|------------|
| 标准场景 | 月底有满 | 月底有满 | - | 1月底满→2月底满 |
| 月初加满 | 无满 | 月初+月底各有满 | - | 2月初满→2月底满 |
| 仅月初满 | 无满 | 月初有满 | 月初有满 | 2月初满→3月初满 |
| 本月无满 | 月底有满 | 无满 | 月初有满 | 1月底满→3月初满 |
| 前后都无满 | 无满 | 无满 | 无满 | 无数据 |

##### 业务规则

- 每次加满既是上一周期的结束，也是下一周期的开始，周期无缝衔接
- 同一加油记录只归属一个周期，不会重复计算
- 月内有多次加满时，最后一次作为周期切割节点，其余仅参与加油量计算
- 月内仅有一次加满时，由上下文决定其作为开始节点还是结束节点

##### 异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| N-1月和N月都无加满记录 | N月无油耗数据，不生成记录 |
| N月无加满且N+1月也无加满 | N月无油耗数据 |
| 车辆长期无加满记录（超过2个月） | 被覆盖的月份无数据，周期归属到最后一个被跨过的月份 |

#### F002：周期内全量加油记录计算

##### 业务规则

满到满周期内，所有加油记录的加油量均纳入该周期"实际升数"计算：

| 记录类型 | 作为周期切割节点 | 纳入加油量计算 |
|----------|-----------------|---------------|
| 周期节点加满记录 | 是 | 是 |
| 非节点加满记录 | 否 | 是 |
| 非满油加油记录 | 否 | 是 |

##### 计算公式

```
实际升数 = Σ 周期内所有加油记录的加油量（升）
平均油价 = Σ 周期内所有加油记录的油费金额 ÷ 实际升数
```

---

#### F003：车辆首条加油记录默认满油

##### 业务规则

- 车辆在系统内最早的加油记录（按时间排序的第一条），默认按满油处理
- 该记录作为该车辆首个满到满周期的起始节点
- 首条记录之后的第一次加满记录作为首个周期的结束节点

---

#### F004：人工修正周期节点

##### 适用场景

| 场景 | 说明 | 操作方式 |
|------|------|----------|
| 有计算结果但不合理 | 当月有满到满周期，但开始/结束时间不符合管理要求 | 直接修改开始/结束时间 |
| 无计算结果 | 当月无满到满周期 | 先在加油记录中补充加满标记或添加加满记录，刷新后再修正 |

##### 交互流程

1. 管理员在月报表中勾选**某月某车**，点击"修改"
2. 系统展示当前满到满周期的**开始时间**和**结束时间**
3. 管理员通过下拉选择新的开始时间和结束时间
4. 提交保存

##### 可选时间范围

下拉可选的时间节点需同时满足：

- **开始时间可选范围**：上月满到满开始时间**之后** ~ 当月满到满结束时间**之前**，且为标记"加满"的加油记录
- **结束时间可选范围**：当月满到满开始时间**之后** ~ 下月满到满结束时间**之前**，且为标记"加满"的加油记录

> **示例**：修改2月，1月周期为12月31日→1月30日，3月周期为2月28日→3月31日
> - 开始时间可选：12月31日之后 ~ 2月28日之前的所有加满记录
> - 结束时间可选：选定的开始时间之后 ~ 3月31日之前的所有加满记录

##### 提交校验

- 所选开始时间必须早于结束时间
- 无计算结果的月份不支持直接修改，需先补充加满记录后刷新

---

#### F005：修正后自动重算

##### 业务规则

- 人工修正保存后，系统自动联动更新**前一月、当月、后一月**共三个月的满到满时间节点及油耗数据
- 重算范围包括：分段明细、标准升数、实际升数、平均油价、奖惩金额
- 因可选范围被上月开始时间和下月结束时间约束，修改影响范围最多三个月，不会继续扩散

---

#### F006：历史数据刷新兼容

##### 业务规则

- 系统已有手动刷新功能，需确保新的节点选取逻辑在手动刷新时生效
- 客户可选择指定月份重新计算
- 刷新后的数据与新逻辑下自动计算的结果一致

---

#### F007：司机离职结算

##### 业务规则

- 司机离职时进行一次加满操作
- 该加满记录作为当前周期的结束节点
- 系统自动计算该周期的标准升数、实际升数和奖惩金额
- 结算数据正确归属到对应月份

---

### 3.3 非功能需求

- **数据一致性**：同一加油记录不会同时归属到不同月份周期，避免重复计算
- **性能要求**：历史数据批量刷新不影响系统正常使用
- **向下兼容**：满到满基本计算模式不变，仅优化月度报表的节点选取逻辑

- 结束时间下拉在选定开始时间后动态刷新可选范围
- 保存后列表自动刷新，展示更新后的数据

## 5. 数据需求

### 5.1 数据模型变更

本次优化主要修改计算逻辑，现有数据表结构基本不变。涉及的核心数据表：

| 数据表 | 变更说明 |
|-------|---------|
| `bs_oil_segment_detail` | 无结构变更，月份归属逻辑调整 |
| `bs_oil_segment_month_driver` | 无结构变更，汇总逻辑调整 |
| `bs_oil_segment_oil_full` | 建议新增字段 `is_manual_modified`（Boolean），标记是否经过人工修正 |
| `common_approval` | 无结构变更，查询范围调整（含非满油记录） |

### 5.2 新增/建议字段

#### bs_oil_segment_oil_full 表

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| is_manual_modified | Boolean | 否 | 是否经过人工修正，默认false |
| modified_by | Integer | 否 | 修正操作人ID |
| modified_time | LocalDateTime | 否 | 修正操作时间 |

### 5.3 数据流转

```
司机加油记录（common_approval）
    ↓ 查询满油记录（tank_full = true）
优化后的节点选取逻辑
    ↓ 确定每月的开始/结束节点
形成满到满周期（bs_oil_segment_oil_full）
    ↓ 在周期内按影响因素分段
生成分段明细（bs_oil_segment_detail）
    ↓ 汇总到月份
生成司机月报表（bs_oil_segment_month_driver）
    ↓ 计算奖惩
油耗奖惩金额
```

## 7. 验收标准

### 7.1 功能验收

#### 自动计算逻辑
- [ ] 标准场景：1月底满、2月底满 → 2月油耗 = 1月底满→2月底满
- [ ] 月初加满场景：1月无满、2月初满+2月底满 → 2月油耗 = 2月初满→2月底满
- [ ] 仅月初满场景：1月无满、2月初满、3月初满 → 2月油耗 = 2月初满→3月初满
- [ ] 本月无满场景：1月底满、2月无满、3月初满 → 2月油耗 = 1月底满→3月初满
- [ ] 前后都无满场景：1月无满、2月无满、3月无满 → 2月无数据
- [ ] 车辆首条加油记录默认视为满油，正确作为首个周期起点

#### 人工修正
- [ ] 有计算结果时，可勾选某月某车点击修改，修改满到满的开始/结束时间
- [ ] 无计算结果时，不支持直接修改，提示先补充加满记录后刷新
- [ ] 下拉可选时间范围正确：开始时间 > 上月周期开始，结束时间 < 下月周期结束，且均为加满记录
- [ ] 提交时校验开始时间早于结束时间
- [ ] 保存后自动联动更新前一月、当月、后一月的数据
- [ ] 修改影响范围不超过三个月

#### 数据完整性
- [ ] 周期内所有加油记录（含非满油）正确纳入实际升数
- [ ] 同一加油记录不会出现在不同月份周期中
- [ ] 奖惩金额按新逻辑正确计算到对应月份

#### 其他
- [ ] 司机离职加满后正确触发结算
- [ ] 手动刷新历史月份时使用新逻辑

### 7.2 性能验收

- [ ] 月报表计算响应时间不超过现有水平
- [ ] 历史数据批量刷新不影响系统正常使用
