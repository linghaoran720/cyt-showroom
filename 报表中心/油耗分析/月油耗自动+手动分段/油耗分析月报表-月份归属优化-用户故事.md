# 用户故事集

## Epic: 油耗分析月报表 - 满到满周期月份归属逻辑优化

---

### Story 1: 系统自动选取月度油耗周期节点

**用户故事**
作为 车队管理员
我想要 系统能根据每月加满记录自动确定油耗计算的开始和结束节点
以便 无需人工干预即可获得合理的月度油耗数据

**优先级**：P0

**故事点估算**：8

**验收标准**
- [ ] AC1: 当N-1月和N月都有加满记录时，N月油耗周期 = N-1月最后一次加满 → N月最后一次加满
- [ ] AC2: 当N-1月无加满但N月有加满时，N月油耗周期的开始节点 = N月第一次加满，同时该节点作为N-1月油耗的结束节点
- [ ] AC3: 当N月无加满但N+1月有加满时，N月油耗周期的结束节点 = N+1月第一次加满
- [ ] AC4: 当N-1月和N月都无加满，或N月和N+1月都无加满时，N月无油耗数据
- [ ] AC5: 月内有多次加满时，最后一次作为周期切割节点，其余仅参与加油量计算
- [ ] AC6: 周期之间无缝衔接，每次加满既是上一周期的结束也是下一周期的开始

**技术说明**
- 涉及模块：`BsOilSegmentMonthDriverService`、`VehicleOilSegmentTaskAsyncMsgService`
- 需修改现有固定取"本月最后满"的节点选取逻辑
- 需兼容现有分段明细计算引擎

### Story 2: 周期内所有加油记录纳入实际升数计算

**用户故事**
作为 财务人员
我想要 满到满周期内的所有加油记录（包括未加满的）都纳入实际升数计算
以便 油耗奖惩数据更准确、更完整

**优先级**：P0

**故事点估算**：5

**验收标准**
- [ ] AC1: 周期节点的加满记录的加油量纳入实际升数计算
- [ ] AC2: 周期内非节点的加满记录的加油量纳入实际升数计算
- [ ] AC3: 周期内非满油加油记录的加油量纳入实际升数计算
- [ ] AC4: 同一加油记录只归属一个周期，不会在不同月份中重复计算
- [ ] AC5: 实际升数 = 周期内所有加油记录的加油量之和

**技术说明**
- 涉及模块：`BsOilSegmentDetailService`、`Approval4OilSegmentServiceImpl`
- 查询在途费用时需包含 `fee_type = ON_WAY_OIL` 的所有记录，不限于 `tank_full = true`

### Story 3: 车辆首条加油记录默认视为满油

**用户故事**
作为 车队管理员
我想要 车辆在系统内的第一条加油记录自动视为满油
以便 新入队车辆也能正常参与油耗周期计算，不会因缺少起始节点而遗漏数据

**优先级**：P1

**故事点估算**：3

**验收标准**
- [ ] AC1: 车辆在系统内最早的加油记录默认按满油处理，作为首个周期的起始节点
- [ ] AC2: 首条记录之后的第一次加满记录作为首个周期的结束节点
- [ ] AC3: 首个周期内的所有加油记录正确纳入实际升数计算

**技术说明**
- 涉及模块：`Approval4OilSegmentServiceImpl`
- 查询满油记录时需特殊处理车辆的首条加油记录

### Story 4: 人工修正满到满周期节点（有计算结果场景）

**用户故事**
作为 车队管理员
我想要 在月报表中修改某车某月的满到满周期的开始和结束时间
以便 当系统自动计算结果不符合管理要求时，能手动调整为合理的周期

**优先级**：P1

**故事点估算**：8

**验收标准**
- [ ] AC1: 在月报表中可勾选某月某车，点击"修改"进入修正界面
- [ ] AC2: 修正界面展示当前满到满周期的开始时间和结束时间
- [ ] AC3: 开始时间下拉可选范围：上月满到满开始时间之后 ~ 当月满到满结束时间之前，且为标记"加满"的加油记录
- [ ] AC4: 结束时间下拉可选范围：当月满到满开始时间之后 ~ 下月满到满结束时间之前，且为标记"加满"的加油记录
- [ ] AC5: 提交时校验所选开始时间早于结束时间
- [ ] AC6: 保存后系统自动联动更新前一月、当月、后一月共三个月的满到满时间节点及油耗数据
- [ ] AC7: 修改影响范围不超过三个月

**UI/UX 说明**
- 月报表列表增加勾选和"修改"操作按钮
- 修正弹窗展示当前开始/结束时间，下拉选择新时间
- 保存前展示影响范围（前一月、当月、后一月）

### Story 5: 人工修正满到满周期节点（无计算结果场景）

**用户故事**
作为 车队管理员
我想要 在某月无满到满计算结果时，通过补充加满记录后刷新来生成数据
以便 即使出现极端情况也能产出该月的油耗数据

**优先级**：P2

**故事点估算**：3

**验收标准**
- [ ] AC1: 当月无满到满计算结果时，不支持直接修改操作
- [ ] AC2: 管理员可在加油记录中将某条记录标记为"加满"，或单独添加一条加满记录
- [ ] AC3: 刷新后系统根据新的加满记录重新计算，产生满到满结果
- [ ] AC4: 产生结果后可按 Story 4 的方式进行进一步修正

**UI/UX 说明**
- 无数据月份给出提示："当月无加满记录，请在加油记录中补充加满标记后刷新"

### Story 6: 修正后自动重新计算油耗数据

**用户故事**
作为 财务人员
我想要 人工修正周期节点后，系统自动重新计算相关月份的标准升数、实际升数和奖惩金额
以便 修正后无需额外操作即可获得正确的油耗报表数据

**优先级**：P1

**故事点估算**：5

**验收标准**
- [ ] AC1: 修正保存后，前一月、当月、后一月的分段明细自动重新计算
- [ ] AC2: 相关月份的标准升数重新汇总
- [ ] AC3: 相关月份的实际升数根据新的周期范围重新汇总
- [ ] AC4: 相关月份的奖惩金额根据新数据重新计算
- [ ] AC5: 满到满汇总表同步更新

**技术说明**
- 涉及模块：`BsOilSegmentMonthDriverService`、`BsOilSegmentOilFullService`
- 需确保重算范围精确为受影响的三个月，不扩散

**用户故事**

以便 历史数据也能按新规则重新计算，保持数据一致性

**优先级**：P1

**故事点估算**：3

**验收标准**
- [ ] AC1: 手动触发指定月份重新计算时，使用新的节点选取逻辑
- [ ] AC2: 刷新后的数据与新逻辑下自动计算的结果一致
- [ ] AC3: 刷新操作不影响系统正常使用

**技术说明**
- 涉及模块：现有手动刷新入口
- 需确保新旧逻辑切换后，刷新功能调用的是新逻辑

### Story 8: 司机离职油耗结算

**用户故事**
作为 财务人员
我想要 司机离职加满油后能自动触发油耗结算
以便 及时完成离职司机的油耗奖惩核算

**优先级**：P2

**故事点估算**：3

**验收标准**
- [ ] AC1: 司机离职时进行加满操作，该加满记录作为当前周期的结束节点
- [ ] AC2: 系统自动计算该周期的标准升数、实际升数和奖惩金额
- [ ] AC3: 结算数据正确归属到对应月份

**技术说明**
- 需与离职流程对接，确保离职加满能触发油耗结算
