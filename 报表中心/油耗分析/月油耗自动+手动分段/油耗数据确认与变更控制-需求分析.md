# 需求分析报告

## 1. 需求概述
- **需求名称**：油耗分析月报表 - 数据确认与变更控制机制
- **需求类型**：新功能
- **需求来源**：业务场景分析（与满到满周期优化配套）
- **优先级建议**：P1高

## 2. 业务背景

### 当前现状

司机新增或修改油费上报时，系统会自动更新油耗分析数据。如果新增或修改的油费是"加满"记录，可能影响满到满的分段周期，导致财务人员之前看到的月报表数据产生变化。目前系统没有数据确认/锁定机制，任何油费变更都会直接影响报表数据。

> 说明：油费上报本身已有独立的审批流程，本需求不涉及油费审批，仅控制油费变更对油耗分析报表的影响。

### 客户痛点

1. **数据不稳定**：财务人员查看的月报表数据可能随时因司机补录/修改油费而变化
2. **已核算数据被覆盖**：财务已基于报表数据完成奖惩核算，但司机后续修改导致数据变化，引发对账混乱
3. **无感知变更**：数据变化时财务人员无感知，可能导致错误的奖惩发放

### 期望目标

- 建立月度油耗数据的"确认"机制，区分已确认和未确认状态
- 未确认的数据允许自动变更（适应司机正常上报节奏）
- 已确认的数据变更时，提醒管理员并由管理员确认后更新
- 变更影响满到满周期时，明确提示影响范围

### 业务价值

- 保护已确认的财务数据，避免被无感知覆盖
- 建立数据变更的可控流程，提升数据治理水平
- 减少因数据变化导致的对账纠纷

## 3. 用户角色与场景

| 角色 | 场景描述 | 核心诉求 |
|------|----------|----------|
| 财务人员 | 每月核算油耗奖惩后，确认当月数据锁定 | 确认后数据稳定，不被随意变更 |
| 车队管理员 | 收到数据变更提醒，确认是否更新已确认数据 | 清楚变更影响范围，可控地决策 |
| 司机 | 补录或修改历史月份的加油数据 | 能正常提交数据 |

## 4. 功能需求拆解

### 4.1 核心功能

#### F1：月度油耗数据确认机制

##### 数据状态定义

| 状态 | 说明 | 数据变更行为 |
|------|------|-------------|
| 未确认 | 月报表计算完成，但财务尚未确认 | 司机新增/修改油费后，系统自动更新报表数据 |
| 已确认 | 财务已核算并确认当月数据 | 司机新增/修改油费时，不自动更新，生成变更提醒 |

##### 确认操作

- 财务人员在月报表中对某月数据执行"确认"操作
- 确认粒度：按月份整体确认（该月所有车辆的油耗数据一起确认）
- 确认后状态变为"已确认"，报表数据锁定

##### 自动变更规则

- **当前月（未结算月）**：数据始终为"未确认"状态，司机油费变更自动更新
- **上月（结算中）**：在未确认之前，油费变更自动更新（因为司机不一定在月底最后一天完成上报）
- **已确认月份**：油费变更不自动更新，生成变更提醒通知管理员

#### F2：变更影响检测与提醒

##### 触发条件

当司机新增或修改油费记录时，系统检测该变更是否影响已确认月份的油耗数据：

| 检测项 | 说明 |
|--------|------|
| 加油时间归属 | 该油费记录所属的满到满周期归属哪个月份 |
| 是否加满 | 如果是加满记录，可能影响满到满周期的分段 |
| 归属月份状态 | 该月份是否已确认 |
| 跨月影响 | 如果影响满到满周期，是否连带影响相邻月份 |

##### 提醒内容

当变更影响已确认月份时，系统提醒管理员：

- 明确告知：哪条油费记录发生了变更
- 影响范围：会影响哪个月份的数据
- 周期影响：如果影响满到满周期，明确告知"确认后会对X月和Y月的满到满数据都产生影响"
- 操作选项：**确认更新** / **取消**（不处理）

#### F3：变更确认流程

##### 流程说明

```
司机新增/修改油费（油费上报已有独立审批流程）
    ↓
系统检测影响范围
    ↓
┌─ 影响未确认月份 → 自动更新报表，无需额外操作
│
└─ 影响已确认月份 → 生成变更提醒
        ↓
    通知管理员/财务
        ↓
    管理员查看变更详情和影响范围
        ↓
    ┌─ 确认更新 → 更新报表数据，相关月份重新计算
    │
    └─ 取消（不处理）→ 报表数据暂不更新，提醒保留待后续处理
```

##### 变更详情展示

| 信息项 | 说明 |
|--------|------|
| 变更人 | 司机姓名 |
| 变更内容 | 新增/修改的油费记录详情（时间、金额、升数、是否加满） |
| 影响月份 | 受影响的已确认月份列表 |
| 周期影响 | 是否影响满到满周期分段，如影响则说明涉及哪两个月 |
| 数据对比 | 变更前后的油耗数据对比（标准升数、实际升数、差额、奖惩金额） |

#### F4：确认撤回

##### 业务规则

- 已确认的月份支持撤回确认，恢复为"未确认"状态
- 撤回后，该月份的待处理变更提醒自动执行更新
- 撤回操作需记录操作人和操作时间

### 4.2 辅助功能

#### F5：变更记录查询

- 记录所有对已确认月份的变更提醒和处理结果
- 支持按月份、车辆、司机查询变更历史

### 4.3 非功能需求

- **实时性**：油费变更后，影响检测和提醒需实时触发
- **数据一致性**：确认更新后的重算逻辑与自动计算逻辑一致

## 5. 边界与约束

### 包含（In Scope）
- 月度数据确认/撤回确认机制
- 未确认数据自动变更
- 已确认数据变更提醒与确认更新流程
- 变更影响检测与提醒
- 满到满周期变更影响提示
- 变更记录查询

### 不包含（Out of Scope）
- 油费上报的审批流程（已有独立流程）
- 部分确认（不支持只确认某几辆车的数据）
- 自动确认（不支持到期自动确认）

### 依赖项
- 依赖"满到满周期月份归属逻辑优化"功能
- 依赖现有油费上报及审批流程
- 依赖现有报表计算引擎

## 6. 风险评估

| 风险项 | 影响程度 | 应对策略 |
|--------|----------|----------|
| 待处理变更提醒堆积过多，管理员处理不及时 | 中 | 定期提醒，提供变更列表集中处理 |
| 长期不处理的变更提醒导致报表数据与实际油费不一致 | 中 | 提供撤回确认功能，撤回后自动更新 |
| 确认后发现数据有误需要修改 | 低 | 提供撤回确认功能 |

## 7. 验收标准

### 数据确认机制
- [ ] 财务人员可对某月数据执行"确认"操作
- [ ] 确认后月份状态变为"已确认"
- [ ] 已确认月份支持撤回确认

### 自动变更规则
- [ ] 当前月和未确认的上月：油费变更自动更新报表
- [ ] 已确认月份：油费变更不自动更新，生成变更提醒

### 变更确认
- [ ] 影响已确认月份时生成变更提醒并通知管理员
- [ ] 提醒信息包含影响范围、数据对比
- [ ] 影响满到满周期时，明确提示会对哪两个月的数据产生影响
- [ ] 确认更新后自动重算相关月份数据
- [ ] 取消不处理时报表数据不变，提醒保留

### 变更记录
- [ ] 可查询所有变更提醒和处理结果
