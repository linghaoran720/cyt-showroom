## 1. 概述

### 1.1 背景

调度与录单人员长期在微信等渠道接收**非结构化文字或截图**任务；需要快速对齐财运通「创建运单」页字段，并支持**一车多趟**等拆合单逻辑。**本文**聚焦运单场景下的 **智能录单助手**（IP **财财**）：**PC Web** 支持 **文本 + 本地文件（图片 / Excel / PDF）** 组合识别；**移动端** 支持 **拍照/相册多图 + 文本** 混合识别；均与线上助手交互对齐，并共用识别与建单管线（见 §7、§10）。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| 文本一键结构化 | 用户粘贴文本或上传 **文件**（PC：图片/Excel/PDF），系统解析为多条运单候选，字段可对照开单页校对 |
| 拆合单一致 | 以 **装+卸+车线+车牌+司机** 五元组划分「运单组」卡片；组内 **建单数量** 支持 ×N/*N 等倍数（见 §4）；与线框 [PT/index.html](PT/index.html) 一致 |
| 个性化短语 | 通过「客户简称扩展」把客户常用的**简称**一键展开为车线、客户、货名等组合信息，减少重复录入 |
| 移动可用 | **App 内嵌页等移动端**提供与现网一致的**图文多模态录入 + 识别 + 人工确认建单**闭环；语音输入后续扩展 |

### 1.3 V1.0 范围

**包含**

- **文本**智能解析（单段内可多运单、多车牌、多趟次）。**PC Web**：通过 **文件** 选择器上传 **图片**（PNG、JPG、JPEG 等常用位图格式）、**Excel**（如 .xlsx、.xls）、**PDF**，可与文本组合提交（见 §6.5）。**移动端**：**拍照 / 相册多选图片** 与文本的 **仅图 / 仅文 / 图文混合** 提交（见 §6.4）。上述均进入同一套识别管线（由 **doubao-seed-2-0-pro**、OCR/表格解析与网关编排协同，见 §10）。
- 运单组拆分（**五元组**见 §1.4）、卡片字段表展示、**建单数量/倍数**（与原文 ×N、*N 等解析一致，默认 1）
- 输出/编辑区字段与财运通「创建运单」页**语义对齐**（见第 4 节字段表）；移动端结果卡字段与现网一致（司机姓名、车牌、押运员电话、主驾手机、货名、装车时间、重量、计划发车、运单号、挂车号等，可随配置裁剪）
- **客户简称扩展**：租户级配置；客户预先维护「简称 → 一段展开说明」，录单时原文出现该简称则按说明带入运单相关字段（详见第 5 节）
- **识别与建单分离**：识别返回结果后 **不自动** 调用创建运单接口；用户点击 **「确定建单」** 后再落库（见 §6.3、§6.4）
- **主数据补全**：用户常提供**片段信息、不完整信息或简称**；在模型/规则抽取得到字段候选后，服务端按 **§3.1、§7.4** 对核心字段查询财运通 **车辆、司机、地址库、车线** 等主数据做**模糊匹配**并**回显标准值**；用户 **确定建单** 后仅调用**既有创建运单接口**，运单创建页内**关联带出、校验链路**遵循现网逻辑，**不在本文展开**
- 与开单流程衔接：确认后逐条或批量写入创建运单（具体接口与现网一致，本文只定义产品行为）

**不包含（可放后续版本）**

- **语音**输入（移动端预留「后续扩展语音」能力位，V1.0 不验收）
- **PC单次极大批量** Excel 行（如数百行）的性能与交互专项、**非运单类**文档模板（若与通用识别平台重复，以平台能力为准）
- 简称扩展规则的批量导入导出（V1.0 可仅支持页面逐条维护，导入导出另议）

### 1.4 术语

| 术语 | 定义 |
|------|------|
| 运单组 | **助手侧一张运单组卡片**对应一条建单勾选行：当 **装货地、卸货地、车线名称、车牌号、司机（主驾）** 五元组经 §3.1 补全后的展示值 **完全一致** 时归为同一组；**任一项不同**（例如不同车牌或不同司机）即拆为不同运单组。**不再**在「同装同卸同货」下把多车合并到同一卡片。货物名称、重量等仍在卡片字段区展示，但不作为与上述五元组并列的「分组主键」。 |
| 建单数量（建单倍数） | 同一运单组拟生成物理运单的张数：默认 **1**；原文 **×N、*N、N 单** 等解析为 **N**，展示在卡片 **右上角（或等价位置）** 且用户可改；与底部 **「确定建单（N单）」** 汇总口径见 §6.3.1 |
| 客户简称（触发词） | 客户在简称配置里使用的**特定字样**，一般为简称；与一条「展开说明」绑定 |
| 展开说明 | 与该简称对应的一段文字或结构化描述，表示一车线、客户、货类、单价等**较复杂信息的打包**；识别命中简称后用于预填运单 |
| 必显字段 | 每条运单输出中无论是否有值都必须展示的字段（见4.1） |
| 主数据补全 | 将识别得到的字段值，与财运通内置主数据（车辆库、司机库、地址库、车线等）做匹配后，替换或填充为**系统标准展示值**的过程；用于助手对话与运单组卡片 **回显** |

## 2. 用户与场景

### 2.1 角色

| 角色 | 诉求 |
|------|------|
| 调度 / 录单（Web） | 在创建运单列表侧打开助手，粘贴群消息，校对后建单 |
| 外勤 / 销售（移动） | 转发或复制客户文字，用手机完成粗解析与确认（V1.0 需具备可落地的移动端方案，具体形态由研发评估） |

### 2.2 典型场景

1. 微信群发来「装/卸/货/车号/司机/运费」混排一段文字 → 解析为多组、多条运单。
2. 消息中含「北京白钢」等客户简称 → 命中简称扩展配置后，按展开说明自动带出车线、客户、品名、单价等。
3. 同装同卸同货，三辆车（不同车牌或不同司机）→ 按 **§1.4 五元组** 拆为 **三张运单组卡片**；若同一卡片内需多物理单，用 **建单数量 N**（×N/*N）表达，而 **不再** 用「同一卡片内多运单子行」表达不同车牌。
4. 司机用手机 **拍运单/聊天截图 + 补充打字** → 提交识别 → 校对运单组 → **确定建单** 落库。
5. 群消息只写 **车牌后五位00123** → 识别后助手回显 **京A00123**（车辆库模糊匹配唯一或取最新）→ 用户确认后建单。

## 3. 核心工作流程（产品行为约定）

以下描述**助手侧**应对用户文本履行的逻辑，与对话式「智能录单」说明一致，并落实在 UI 与接口契约中。

1. **简称扩展预处理**：对原文做客户简称匹配；命中则按配置的展开说明合并进识别上下文或直接写入字段候选（冲突时以原文显式字段优先，规则见 5.2）。
2. **分组（助手卡片 = 运单组）**：以 **装货地、卸货地、车线名称、车牌号、司机（主驾）** 五元组为键（均取 §3.1 补全/回显后的标准展示值；车线名称含简称扩展命中后的 `route_name`），将识别结果归为若干 **运单组**；五元组一致合并为 **一张卡片**，任一项不同则 **拆成多张卡片**。
3. **建单数量**：在同一运单组内，根据原文 **×N、*N、N 单** 等解析 **建单数量 N**（默认 **1**），与向创建运单接口提交的物理单数一致；**不再**在同一卡片下用「运单 i（车牌，N 趟）」多行并列展示不同车牌——不同车牌已在上一步拆组。
4. **字段填充**：为每条目填充必显字段与可选字段；必显字段无值时显示为空或「未提供」（由 UI 统一文案）。
5. **输出头**：解析结果顶部必须展示：**「本次共解析出 X 条运单，分为 Y 个运单组」**（**X** = 各卡片 **建单数量** 之和，即物理运单总数；**Y** = 运单组卡片张数，与 §1.4 五元组分组一致）。
6. **发起识别（仅解析，不建单）**：用户通过 **PC「发送」**（§6.5）或 **移动端「提交」**（§6.4）提交 **文本与/或附件**（PC 含图片/Excel/PDF，移动侧以图片为主）；服务端完成抽取后执行 **§3.1 主数据补全**，再向前端返回运单组卡片（回显为标准值）。**本步骤不得调用创建运单接口**。（识别中态见 §6.4.5、§6.5.3。）
7. **人工校对**：用户 **仅** 通过运单组卡片 **「编辑 >」** 进入 **就地编辑**（字段切换为可输入态，视觉焦点不跳出卡片），**不再**提供「表单修改（弹窗）」等第二套校对入口。可调整各组 **建单数量**（右上角数字，默认来自解析）。可对本轮识别体验提交 **「吐槽」**（见 **§6.6**）。
8. **提交建单**：用户在 **识别结果区域最底部** 点击 **「确定建单（N单）」**（**N** = 当前 **已勾选** 且 **尚未建单成功（未锁定）** 的运单组 **建单数量** 之和），系统 **仅对本次勾选范围内的运单组** 发起创建运单请求（已成功建单的组 **不得** 再次上送）。**不得**将主按钮放在单张运单组卡片内（与现网线框一致）。
9. **建单过程反馈（单气泡）**：用户点击 **确定建单** 后，助手在 **同一条会话气泡内** 先展示 **本次规模**（**K** 个运单组、**P** 张物理运单）与 **「正在建单…」**。气泡内展示 **「执行进度」**（横向进度条 + **已完成数/总数**，如 `0/7` → `7/7`）。**本批次** 未结束前 **不** 再单独追加一条「仅进度」气泡。
10. **建单结果、锁定与重提**：**本批次** 建单 **结束后**，**仍在步骤 9 同一条助手气泡内** 切换/接续为 **按运单组** 的成功或失败块（多组时 **多块上下排列**），**不再**为「仅结果」再推一条独立气泡（减少会话条数、便于翻阅历史）。**成功**组：卡片进入 **已建单锁定态**——**去掉勾选框与建单数量编辑、去掉「编辑 >」**，**不可**再纳入 **「确定建单」** 提交范围。**失败**文案含 **「点击修改」**：**自动仅勾选该失败组**（同轮其他未锁定组取消勾选）；该组 **识别卡片从原识别结果区移除**，**插入到该条失败气泡正下方**（迁入区含说明、可编辑卡、**本区「确定建单」**），用户 **无需滚回** 顶部原识别区；**原识别区底部「确定建单」的 N 不含已迁出组**。点击 **「确定建单」**（迁入区或原区）时 **须** 将当前输入区内未点「完成」的改动 **一并作为提交内容**。迁入组 **建单成功后**：**取消迁出态**，卡片 **回到原识别结果区** 以 **已建单锁定** 展示，失败气泡下迁入区 **消失**。用户仍可对其余 **未迁出** 的失败组 **滚动回原识别区** 勾选编辑后使用 **原区确定建单**。**本轮全部运单组均建单成功** 后，**隐藏** 原区 **「确定建单」**（**「吐槽」** 等仍可保留）。**不得**要求用户重新粘贴原文（除非主动清空会话）。

### 3.1 识别结果的主数据补全与回显

用户输入可能是 **部分字段、独立片段、简称或后几位**（如车牌只写 `00123`）。在 **大模型/规则完成字段抽取**（及客户简称扩展，见 §5）之后、**向前端返回运单组卡片之前**，服务端须对下列 **核心字段** 执行 **财运通主数据匹配与补全**，并在 **助手对话、运单组卡片（含就地编辑初始值）** 中统一使用 **补全后的标准展示值**。

**3.1.1 字段与数据源、匹配顺序**

| 识别侧字段（示例） | 主数据查询来源 | 匹配方式（产品约定） |
|--------------------|----------------|----------------------|
| **车牌号** | **车辆信息管理** | **模糊查询**（支持用户输入片段与系统全量号段匹配，如 `00123` 命中 `京A00123`）；命中 **0 条** → 保留模型输出原文，并在助手内 **明确提醒用户**：**「未查询到车辆」**（见 §3.1.4）；命中 **1 条** → 带入该车辆档案中的**标准车牌展示**；命中 **多条** → 带入 **「最新」一条**（**最新** = 以车辆档案中可排序的业务时间为准，如最近更新时间/最近使用/创建时间，**由研发与业务在方案中选定一种并写死**，本 PRD 不展开表结构） |
| **司机姓名**、**司机电话**（主驾等） | **司机信息管理** | 模糊匹配；命中 **0 条** → **提醒**：**「未查询到司机」**（见 §3.1.4）；**唯一** → 带入标准姓名/电话；**多条** → 带入 **最新** 一条（「最新」定义同上，可与车辆侧一致） |
| **装货地址**、**卸货地址**（及解析得到的地址文本） | **先地址库 → 再车线管理** | **先**在 **地址库** 中做匹配/归一化得到标准地址表述；**再**按需匹配 **车线管理**（如带出车线名称、里程等现网已有能力）；具体算法与优先级由研发对接现网模块 |
| 其他字段 | — | 不在本表强制要求；若后续扩展主数据表，以变更记录追加 |

**3.1.2 回显规则（对用户可见）**

- 卡片与气泡中展示的内容以 **补全后** 为准。示例：用户输入 `00123`，匹配到车辆 `京A00123`，则 **车牌号** 展示为 **`京A00123`**（而非保留用户原文）。
- **未命中主数据**时：在运单组卡片或对话区展示 **§3.1.4** 规定的提醒文案，避免用户误以为已建档。
- 可选（产品增强）：对发生过补全的字段展示 **提示角标/文案**（如「已匹配车辆档案」），V1.0 不强制，由 UX 决定。

**3.1.3 与「确定建单」的关系**

- 主数据补全仅影响 **助手内展示与编辑初始值**；用户点击 **「确定建单」** 后，请求体进入财运通 **既有创建运单接口**。
- 创建运单页面/服务内部的 **复杂关联带出、校验、默认赋值** 等逻辑 **完全遵循现网**，本文 **不要求** 在助手侧重复实现或文档化；若接口需传 **主数据主键 ID** 而非展示文本，由研发在方案中定义助手出参与开单入参的映射。
- **自动新增车辆/司机**：若用户此前 **未查询到车辆/司机**，其在助手中 **补全车牌号、司机姓名或电话** 等信息后点击 **「确定建单」**，**是否**在运单创建过程中 **自动写入车辆信息管理 / 司机信息管理**（建档），**完全以财运通现网创建运单接口与后台配置为准**；助手侧须传递用户最终确认的字段值，**不得**在 PRD 层另造一套建档接口。若现网已支持「无档案自动新增」，则产品验收为：**补全后建单成功且档案侧出现对应记录**；若现网不支持，则验收为：**提示清晰 + 建单失败原因与现网一致**（或引导用户先去主数据维护）。

**3.1.4 未命中时的提醒文案（强制）**

| 场景 | 对用户展示（可微调标点，语义须一致） |
|------|--------------------------------------|
| 车辆库 **0 条** 命中 | **未查询到车辆**。请补全车牌号等信息；补全后提交建单，系统将按现网规则处理，**支持时可在运单创建过程中自动新增车辆档案**。 |
| 司机库 **0 条** 命中 | **未查询到司机**。请补全司机姓名或电话等信息；补全后提交建单，系统将按现网规则处理，**支持时可在运单创建过程中自动新增司机档案**。 |

- 若 **车辆与司机均未命中**，可同时展示两段提醒，或合并为一条 **「未查询到车辆、未查询到司机」**（由 UX 定稿，须覆盖两类主体）。
- 提醒位置：**运单组卡片内显著区域**（如车牌/司机字段下方）或 **助手气泡** 紧随识别结果，确保用户 **发送前/建单前** 可见。

## 4. 财运通智能录单 · 拆合单与输出规则

### 4.1 总原则

- **运单组（卡片）划分**：以 **装货地、卸货地、车线名称、车牌号、司机（主驾）** 五元组判定；五元组一致则为 **同一运单组**，否则拆为 **多张卡片**。车线名称以主数据/简称扩展回显为准，与「装→卸」默认车线可并存，以协议字段 **`route_name`** 为准。
- **建单数量**：同一运单组对应 **一行勾选** 与 **一个建单数量**（默认 1）；原文 **×N、*N、N 单** 等解析为 **N**，与用户可编辑值一致后参与底部 **「确定建单（N单）」** 汇总。
- 输出开头必须标注：**本次共解析出 XX 条运单，分为 XX 个运单组**（XX 条 = 各组建单数量之和；XX 个组 = 卡片数）。

### 4.2 运单拆分与分组规则

| 规则 | 说明 |
|------|------|
| 五元组一致 | **装、卸、车线、车牌、司机** 均相同 → **同一运单组**（一张卡片） |
| 五元组任一项不同 | 拆为 **不同运单组**（例如不同车牌、不同司机各一张卡片） |
| 建单数量 | 默认 **1**；**×N、*N、N 单** 等解析为 **N**，展示在卡片 **右上角（或产品统一位置）** |
| 司机/电话展示 | **司机姓名** 与 **主驾司机手机号** 在卡片 **分列展示**（两列表格/与现网创建运单页字段对齐），**不得**合并为单一「司机&电话」文案展示（就地编辑时亦按字段拆分） |
| 未写趟数/倍数 | 建单数量默认 **1** |

### 4.3 字段输出规则

**必显字段**（无论是否为空均展示，与创建运单页对照录入）

| 对外输出名 | 与开单页对应关系（说明） |
|------------|---------------------------|
| 预计发车时间 | 客户信息区计划/预计发车类时间；仅日期无时刻时保留日期并可在副文案标明「时刻未提供」 |
| 车线名称 | 默认 **装货地→卸货地**；若客户简称扩展或映射规则指定车线则以规则为准 |
| 装货地 | 对应 **起**（高德/详细地址） |
| 卸货地 | 创建运单页「客户信息」中与卸货/终点对应的高德或详细地址 |
| 车牌号 | **运力信息 · 车牌号**；用户提供不完整时原样展示（如 `00123`） |

**可选字段**（仅当解析结果中存在该信息时展示）

| 对外输出名 | 与开单页对应关系（说明） |
|------------|---------------------------|
| 司机姓名 | **运力信息 · 主驾司机**（若文本仅一位司机且无副驾描述，默认主驾）；卡片上与 **主驾司机手机号** 分开展示 |
| 主驾司机手机号 | 运力信息，有则展示；与 **司机姓名** 分列 |
| 副驾司机姓名 | **运力信息 · 副驾**；**仅当** 识别结果中存在该字段有效值时，在卡片（含就地编辑）中展示 |
| 副驾司机手机号 | 同左，有则展示 |
| 押运员电话 | 运力信息，有则展示 |
| 装车时间 | 与 **预计发车时间** 区分；有则展示 |
| 运单号 | 文本/附件中若出现则展示 |
| 挂车号 | **挂车牌号** |
| 货物名称 | **货物信息** |
| 货值 | 货物或备注中有则展示，金额保留两位小数 |
| 重量 | **货物信息 · 重量**等，有则展示 |
| 件数 | 有则展示 |
| 运费 / 应收 | **收入运费 · 运输费** 等，有则展示为 `XXXX.00` |

其他开单页字段（客户名称、项目、网点、支出运费、付款方式拆分等）**在 V1.0 文本中若出现则同样解析展示**；未在消息中出现则不强行展示（与「可选字段」原则一致）。

### 4.4 输出格式标准（示例结构）

```
本次共解析出 {条数} 条运单，分为 {组数} 个运单组（分组键：装+卸+车线+车牌+司机）

运单组 1｜建单数量 {N}
装货地 / 卸货地 / 车线名称 / 车牌号 / 司机姓名 / 主驾手机 / …（与卡片字段表一致）
…

运单组 2｜建单数量 …
…
```

产品验收时以**结构化 JSON** 与 UI 双向一致为准，上述文本为调度员可读摘要形态。

## 5. 客户简称扩展

### 5.0 命名与本质（为何不用「代号翻译」）

**本质**：客户（或本企业录单规范）可以预先维护一些**特定字样**，绝大多数场景下是**简称**；每个简称背后对应**一段较长的、结构可能较复杂的说明**（例如固定车线、固定客户、固定品名与单价口径等）。在后续日常录单中，只要粘贴或输入的原文里**出现该简称**，系统应能**识别到**，并**根据已保存的说明**，把对应信息**带入运单**（或进入识别上下文再抽取字段）。

文档中统称该能力为 **「客户简称扩展」**（简称 = 触发词，扩展 = 按说明展开到运单）。历史上若界面曾使用「代号翻译/解释」等说法，含义与本文一致；**建议新版界面统一为「客户简称扩展」或「简称与展开说明」**，降低理解成本。

### 5.1 功能说明

提供租户级配置能力：维护多组 **客户简称（触发词）** 与 **展开说明**（长文本或结构化字段集合，可由实现约定格式）。智能识别时若原文包含某简称，则将对应展开说明**合并**进当前识别上下文，用于预填车线、客户、货物名称、单价等。

配置弹窗列表列建议：**操作**、**序号**、**简称**、**展开说明**（与现网「代号 / 翻译·解释」列一一对应时，仅改表头文案即可）。支持添加、删除、确定、取消。

### 5.2 业务规则

1. **命中范围**：全角半角、连续出现或作为独立词出现均可配置匹配策略（V1.0 至少支持**完整短语包含匹配**；模糊匹配另议）。
2. **与原文冲突**：原文已明确写出车线、客户、运费等时，**以原文为准**，简称扩展**仅补充空缺项**。
3. **多简称命中**：按配置顺序或最长匹配优先（产品默认 **最长匹配优先**，避免子串误伤）；冲突字段由用户在 **卡片就地编辑** 中最终确认。
4. **权限**：配置入口建议仅管理员或指定角色；普通录单员仅使用识别结果。
5. **审计**：简称扩展配置的增删改记录操作人、时间，便于追溯。

### 5.3 与「映射规则」关系

| 能力 | 侧重点 |
|------|--------|
| 映射规则（通用识别平台） | 识别词 → 系统字段 ID 的通用映射 |
| 客户简称扩展（本文） | 客户/项目级 **简称 → 一包业务信息**（自然语言或键值），偏个性化、业务口径强 |

两者可同时存在：建议解析管线为 **简称扩展合并 → 大模型/规则抽取 → 字段映射**。

### 5.4 识别配置（简称配置 + 映射配置）

PC Web 将 **客户简称扩展** 与 **字段/映射能力** 合并为 **单一入口「识别配置」**，以弹窗（或大抽屉/全屏页）承载 **两个顶层 Tab**，避免底栏多按钮并列。与 **通用智能识别平台** 的 **F004 / F005 / F010** 及 **§5.1～§5.3 简称扩展** 规则同源，细则以 **[2604-智能识别平台-PRD-v1.0.md](../2604-智能识别平台-PRD-v1.0.md) §3.2** 与 **[知识库-智能识别系统.md](../知识库-智能识别系统.md#84-智能录单--字段映射f004--f005--f010-索引)** 为准。

**5.4.1 入口（PC Web · §6.5）**

- 底部工具条 **仅保留一个配置类主入口**：**「识别配置」**（历史若分为「客户简称扩展」「映射管理」两键，本产品迭代 **合并** 为本入口）。
- 点击后打开 **识别配置** 弹窗，内含 **Tab1：简称配置**、**Tab2：映射配置**。**确定/取消** 行为：至少 **简称配置** 的编辑随 **确定** 落租户配置（与 §5.1 一致）；**映射配置** 的保存与 **映射规则库** 写入与 **F010**、系统设置主路径 **同一套后端**。

**5.4.2 Tab1：简称配置**

- **顶部说明（必选展示）**，向用户解释简称与展开说明的用途。产品推荐文案要点：**长期固定业务** 可维护 **特定简称**（左）与 **展开说明**（右，如客户/项目/装卸地等）；用户在对话中 **提到该简称** 时，助手将 **右侧信息合并进识别上下文**，快速辅助建单。具体标点与示例可由 UX 微调，语义须一致。
- 列表与交互仍遵循 **§5.1、§5.2**（操作、序号、简称、展开说明、添加/删除、确定、取消）。

**5.4.3 Tab2：映射配置**

- **顶部说明（必选展示）**，向用户解释 **识别侧说法 → 系统字段** 的 N→1 关系及 **表格列标题** 场景。产品推荐文案要点：左侧配置如「车」「车牌」、右侧选系统字段「车牌号」时，**导入表格列标题** 为「车」「车牌」等则 **自动映射** 到「车牌号」；**导入时** 系统自动判断映射；**保存识别结果或确认映射时** 可按规则 **自动更新映射表**（与 **F004** 及平台约定一致，冲突走 **F005**）。
- **智能映射过滤（标题 + 内容）**：自动学习/自动落库映射规则时，须结合 **列标题（或识别字段名）与单元格/正文内容** 判断语义是否成立；对 **「12345测试」** 等与 **业务语义、列内容无关联** 的噪声映射 **不自动写入规则**，**除非用户在本 Tab 或等价界面手动添加**。避免脏规则污染全集团映射表。
- **统一映射表（推荐定稿）**：**不再**拆成「N→1 配置区」与「规则列表」两块；**一张表**承载 **F010** 规则维护与 **F004** 所涉多别名 → 单系统字段关系。列建议：**（1）识别字段**——同一系统字段下可维护 **多个识别侧别名**；用户点击 **+** 输入文案后 **回车** 添加别名，**鼠标悬停** 某一别名可 **删除该别名**（不影响同条其他别名）。**（2）系统字段**——**下拉选择**，选项与 **创建运单 · 运单模板配置** 字段 **同源**（字段 ID/名称一致）。**（3）使用次数**——数值文案（如「156 次」），**不使用**进度条等装饰。**（4）最近使用**——日期或时间。**（5）删除**——删除 **整条映射关系**（该行所有别名 + 系统字段绑定）；点击后 **二次确认弹窗**，文案含 **「删除后不可恢复」**，**确认** 执行删除，**取消** 关闭弹窗。表头可对 **（1）（2）** 附简短说明。可保留 **搜索/筛选、分页、添加空行** 等 **F010** 能力；线框见 [PT/index.html](PT/index.html) **映射配置** Tab。

**5.4.4 能力摘要（与平台 PRD 对齐）**

| 项 | 编号 | 要求摘要 |
|----|------|----------|
| 智能字段映射 | F004 | 见平台 PRD；**自动落规则** 须遵守 **§5.4.3** 标题+内容过滤 |
| 映射冲突处理 | F005 | 与全局规则冲突时 **仅本次生效** / **更新映射规则** 二选一，须用户确认 |
| 映射规则管理 | F010 | 列表、统计、删除、搜索；**租户/集团隔离**；与 **系统设置 → 智能识别 → 映射规则管理** 同源 |

**5.4.5 移动端**

- **V1.0**：**不设**与 PC 完全一致的底栏「识别配置」时，简称与映射维护 **默认走** 系统设置/平台主路径（见 **§5.4.1** 例外说明）；若后续统一弹窗，另起需求。

## 6. 交互要求（对齐现网智能录单助手）

### 6.1 Web / 移动端侧栏（识别结果区）

- 展示多张 **运单组**卡片，划分规则见 **§1.4、§4.1**（装+卸+车线+车牌+司机）。卡片主体为 **两列表格**（或现网等效布局），字段含：**司机姓名**、**车牌号**、**重量**、**货物名称**、**主驾司机手机号**、**押运员电话**、**装车时间**、**计划发车时间**、**运单号**、**挂车号**、**运费/应收** 等（以线上为准，可裁剪）。**司机姓名** 与 **主驾司机手机号** 须 **分列**，不得合并为一格展示。
- **车牌、装/卸、车线等** 须展示 **§3.1** 主数据补全后的标准值；**车辆或司机未命中** 时须在卡片显著区展示 **§3.1.4** 提醒。
- 每组卡片：**左侧勾选**（参与底部批量建单）、**右上角建单数量**（默认 1，解析 ×N/*N 后为 N，可编辑）。
- **「编辑 >」**：点击后 **当前卡片就地进入编辑态**（字段变为可输入控件），按钮文案变为 **「完成」**；再次点击 **「完成」** 保存并回到只读态，焦点不离开会话区（不强制整页跳转）。**本产品不提供** 「表单修改（弹窗）」等第二入口；校对规则见 **§6.2**。
- **「确定建单」主按钮不在卡片内**：须放在 **本条识别结果全部卡片之后、会话区底部固定区域**，样式与现网一致（如 **「确定建单（N单）」**，N 为已勾选各组 **建单数量** 之和）；**同一区域** 提供 **「吐槽」** 入口（见 **§6.6**），**不再**提供点赞/点踩。
- **底部录入区（文本 / 文件 / 发送）** 的 **移动端** 见 **§6.4**，**PC Web（财财）** 见 **§6.5**。

### 6.2 运单组卡片 · 就地编辑（唯一校对入口）

本产品 **不提供** 「表单修改（弹窗）」「识别后-修改」「建单失败-修改」等 **独立编辑弹窗**。**识别后校对** 与 **建单失败后改错** 共用 **同一套卡片就地编辑**（与 **§6.1** 两列表格字段一致，以现网助手可展示列为准）。

**6.2.1 入口与操作**

- **识别后**：用户点击卡片 **「编辑 >」** → 进入编辑态（字段格内由只读切换为输入控件），按钮文案变为 **「完成」**；再次点击 **「完成」** → 写回当前组数据并退出编辑态。**不在卡片内** 提供 **「确定建单」**（建单仅 **§6.3.1** 底部按钮）。
- **建单失败后**：会话气泡 **「点击修改」** → **同轮识别结果内仅勾选该运单组**（其他未锁定组取消勾选）；该组卡片 **从原识别区摘出**，**贴在当前失败气泡下方**（迁入区：**说明 + 就地编辑 + 本区「确定建单」**），原识别区 **不再展示该卡**；视区 **自动滚至迁入区**。若接口返回可定位字段，**标签与输入框** **错误色（如红色）** 高亮；**失败原因摘要条** 展示在卡上。用户可 **不点「完成」直接点迁入区「确定建单」**（以提交瞬间输入为准），或先 **「完成」** 再建单。成功后卡片 **回到原识别区锁定**，迁入区关闭（见 **§3 步骤 10**）。**边界（历史气泡）**：会话流为 **只追加、不改写** 时，**较早** 的失败气泡仍显示 **「点击修改」**，但该运单组可能已在 **后续批次** 中建单成功（卡片已为 **锁定态**）。用户点击 **该历史链** 时，须以 **当前运单组是否已锁定** 为准：**已锁定** 则 **Toast** **「运单组已建单完成，不能再修改。」**，**不** 迁入、**不** 打开编辑；**未锁定** 则仍走迁入编辑。**产品示意**见文首配图占位 `reference/智能录单-历史失败气泡-已建单成功误点点击修改.png`（与 **Toast** 文案一致；**不要求** V1.0 自动订正或隐藏历史气泡内链，可由后续迭代优化）。

**6.2.2 提醒与字段范围**

- **主数据未命中（§3.1.4）**、**前端校验失败**、**建单接口字段/业务错误**：须在卡片 **表格上方或显著区** 展示 **摘要提示**（可多条）；无待处理事项时可不展示。
- 卡片展示的字段集合可为现网 **裁剪子集**（与 **§6.1** 一致）；未尽字段若必须校对，由业务在方案中扩展卡片列或跳转创建运单页（本需求 **V1.0** 以助手内就地编辑为主路径）。

**6.2.3 静态线框（演示）**

- 本地 **PT** [PT/index.html](PT/index.html)：**无** 运单组编辑弹窗；**双端** 就地编辑；**确定建单** 为 **单条助手气泡**：约 **2 秒** 进度后 **同气泡** 切换为按组成功/失败（**无**「好的，财财收到」）；**首轮建单** 演示：运单组 **1、4** 成功并 **锁定**，**2、3** 失败；**点击某组「点击修改」** 后该卡 **迁至对应失败块下** 并出现 **迁入区确定建单**；**全部成功后** 隐藏原区 **确定建单**；**识别配置** 见 **§5.4**。
- **PC Web** 线框 **不** 设单独左侧「运单组·识别后展示卡片」栏；运单组仅在 **会话/识别结果流** 中展示。**移动端** 仍为单栏会话 + 底部录入。

### 6.3 提交与建单结果反馈（识别之后）

本节补齐「识别完成 → 落库建单」的交互，与现网 **智能录单助手**（含对话区 + 进度）一致。

**6.3.0 识别与建单两阶段（强制）**

- **识别/解析**：用户提交内容后，系统返回 **运单组卡片**（可多组），仅展示结构化结果，**不**在此阶段自动调用 **创建运单** 接口。
- **建单**：用户核对并 **勾选** 运单组（**不含** 已建单成功的 **锁定组**）、必要时 **就地编辑**（**§6.2**）后，点击 **识别结果区最底部** 的 **「确定建单（N单）」**，才发起创建运单请求。**N** = 所勾选且 **未锁定** 组的 **建单数量** 之和；请求 **仅包含本批勾选组**，不包含已锁定组。**移动端与 Web 行为一致**；**主按钮不得仅出现在单张卡片内**。

**6.3.1 提交入口（建单侧）**

- 在 **识别结果区域最底部** 提供主按钮：**「确定建单（N单）」**，**N** = 当前 **已勾选** 且 **未锁定** 且 **仍留在原识别区（未因失败而迁出）** 的运单组 **建单数量** 合计；**未勾选任一组或 N=0** 时按钮禁用或点击后提示先勾选。**本轮识别结果下全部运单组均已建单成功** 时 **隐藏** 该按钮（**「吐槽」** 等可保留）。
- **迁出重提**：对 **已迁至失败气泡下** 的运单组，在 **迁入区** 提供 **第二处「确定建单」**，语义与原区一致，**仅针对当前迁入组（或迁入区勾选策略与研发对齐）**，无需用户返回原识别区底部操作。
- *不再要求*在卡片内提供 **「确定建单 ×N」** 作为主路径；若现网仍有历史入口，以本产品 **底部统一按钮** 为验收口径。

**6.3.2 提交后的即时反馈（进行中）**

- **单条助手气泡（与 §3 步骤 9 一致）**：**不**再使用 **「好的，财财收到！」** 等寒暄；首屏文案须含 **运单组数 K**、**物理运单总数 P**（与解析头「条数」口径一致）及 **「正在建单…」**（或等价状态）。
- 同气泡内展示 **「执行进度」**：
  - **横向进度条**（建议 **约 2 秒** 内有明确 **从少到满** 的过程感，提升可预期性；实网可与轮询/推送节奏对齐，并设 **最短展示时间** 避免闪烁）；
  - 数字 **已完成 / 总数**（如 `0/7` 递增至 `7/7`）；与后台实际处理同步；部分失败时 **总数** 仍为 P，**已完成** 口径由研发与现网统一（「成功数/总数」或「已处理数/总数」），产品要求 **用户能判断何时全部结束**。

**6.3.3 结束态反馈（成功 / 失败）**

- **仍在 §6.3.2 同一条助手气泡内** 切换或接续为 **按运单组** 的结果块（非每条结果单独再占一条新气泡），与现网 **财财** 体验目标一致（少干扰、易翻历史）。
- **成功**（**本批次** 内多块 **上下排列** 于 **同一条助手气泡**，与 **§3 步骤 10** 一致；文案须含）：**「运单组{序号}，创建成功：」** 换行列出每张物理单的 **「****运单编号****」**（一行一个编号，编号样式与现网一致）。对应运单组卡片进入 **锁定态**：**无勾选、无编辑入口、无再次建单**，展示 **已建单** 等明确状态（与 **§6.2、§3 步骤 10** 一致）。
- **失败**：**「运单组{序号}，创建失败：**{接口返回的报错原因}**，点击修改」**；其中 **「点击修改」** 为可点链，行为见 **§6.2.1**、**§3 步骤 10**（**卡片迁入失败气泡下**、**迁入区确定建单**、成功后 **回到原区锁定**）；**可先「完成」编辑也可直接建单**（以提交瞬间表单内容为准）。**历史失败气泡**：若该组 **此后已成功建单并锁定**，再点旧气泡上的 **「点击修改」** 仅 **Toast** 拦截，见 **§6.2.1 边界**。
- 失败组若在侧栏仍有卡片态，可增加 **「建单失败」** 角标等，避免与待提交混淆（可选，与现网统一）。

**6.3.4 异常与并发**

- 建单进行中应 **禁用重复提交** 或合并为同一任务队列，避免重复落单。
- 网络超时、部分接口失败时，进度区应进入 **可结束态** 并提示用户 **重试或联系管理员**，失败组可走6.3.3 修正链路。

### 6.4 移动端 · 录入与识别交互（图文多模态）

以下与产品原型一致：**从入口进入空白页 → 底部组合输入 → 提交识别 → 结果卡展示 → 用户确认建单**（见文首配图占位「移动端-输入与提交」「移动端-识别结果」）。

**6.4.1 入口与初始态**

- 用户点击功能入口，进入 **空白内容页**（无默认运单组、无历史气泡干扰）。
- 页面 **底部** 为固定 **输入条区域**：左侧 **相机图标**（唤起拍照/选图入口）、中间 **文本输入框**（占位如「在此处粘贴或输入运单信息」）、根据状态在右侧显示 **提交**控件（见 6.4.5）。

**6.4.2 拍照与相册**

- **拍照**：用户通过底部能力选择 **相机**，进入系统相机取景；**完成** 后返回当前页，图片进入待提交列表。
- **相册**：选择 **相册**，进入系统相册，支持 **多选**；**完成(n)** 带回 **n** 张图至当前页。
- 已选图片在输入区 **上方** 以 **横向缩略图** 展示：每张缩略图右上角 **「×」** 可删除；列表末尾 **「+」** 可继续加图（等价于再次走相册/拍照流程）；亦可再次点击 **相机图标** 追加拍摄或选图。

**6.4.3 文本与键盘**

- 点击输入框 **唤起系统键盘**，支持多行输入（具体是否 `Shift+回车` 换行随系统实现）。
- **语音输入** 不在 V1.0 验收范围，可在文案或规划中预留「后续扩展语音」。

**6.4.4 提交控件显隐**

- 当且仅当满足以下 **任一** 条件时，显示 **提交**图标（如输入条右侧 **向上箭头** 按钮）：**至少一张待提交图片**、**输入框内存在非空文本**、或 **图文同时存在**。
- 无任何图片且文本为空时，**不显示** 提交按钮。

**6.4.5 提交识别与「识别中」态**

- 用户点击 **提交**，将 **当前全部待提交内容**（0～多张图片 + 当前文本，允许仅图、仅文、图文混合）发往 **识别/解析** 服务，进入 **识别中** 状态。
- **识别中** 期间：
  - **输入框**：保持 **可编辑**，用户可继续输入文字（例如补充说明）；是否合并进 **下一次** 识别请求由产品定义，**默认建议**：当前轮次仅处理点击提交瞬间的快照；用户在识别中又输入的内容留待下一轮提交（避免与进行中请求打架），若研发实现为「队列合并」需在方案中说明。
  - **相机图标**、**提交图标**：**禁用**（置灰或不可点），直至 **本轮识别结束**。
- 识别 **成功或失败** 结束后，恢复相机与提交可用；用户可继续 **加图 / 改字 / 再次提交** 发起新一轮识别。

**6.4.6 识别结果展示（不自动建单）**

- 助手以对话或卡片形式展示 **「识别结果如下：」**，下方为 **运单组** 卡片（可多组），与 Web 侧 **§6.1** 一致：**五元组分组**、**勾选**、**右上角建单数量**、**两列表格**（**司机姓名** 与 **主驾司机手机号** 分列）、**「编辑 >」/「完成」就地编辑**（**§6.2**，无弹窗第二入口）；在 **全部卡片之后** 展示 **「吐槽」** + **「确定建单（N单）」**（N 为已勾选组建单数量之和），规则见 **§6.6**。
- **禁止** 在识别结果返回后 **自动** 调用创建运单接口；必须由用户点击底部 **「确定建单（N单）」** 后，才进入 §6.3 的建单与进度反馈流程。

**6.4.7 与 Web / 接口**

- 识别与建单 **API** 宜与 Web 端 **复用**；识别入参为 **文本 + 附件列表**（见 §7.2）。
- **交付物**：方案设计阶段明确 **入口位置**（App 内路径）、**最大图片张数/单张大小**、**是否压缩上传**、弱网与失败重试、离线草稿是否要做。

### 6.5 Web PC · 「财财」底部录入区（文件 + 文本 + 发送）

与产品界面一致：**大号多行输入框 + 底部工具条**（见文首配图「PC 端财财」）。

**6.5.1 布局与文案**

- 输入框占位示例：**「在这里输入运单描述信息，shift + 回车 = 换行」**（与现网一致；具体文案可配置）。
- 工具条左侧：**「识别配置」** 单一入口（**§5.4**），打开弹窗：**Tab1 简称配置**（原客户简称扩展/代号翻译语义）、**Tab2 映射配置**（F004/F005/F010，与系统设置映射规则 **同源**）。**不再**在底栏并列「客户简称扩展」+「映射管理」双按钮（历史过渡期除外，须在方案中声明下线）。
- 工具条中部：其它能力位（如使用示例等）保持与现网一致。
- 工具条右侧：**文件** 按钮 + **发送** 按钮（纸飞机图标 +「发送」）。

**6.5.2 文件入口（替代原「图片」图标）**

- **不得**再使用单独的「图片」图标作为主入口；统一为 **「文件」** 图标（或「附件」），点击后打开 **本地文件选择**（系统文件对话框）。
- **可选类型**（需在 `accept` 或等价过滤中体现，具体扩展名以研发白名单为准）：
  - **图片**：PNG、JPG、JPEG 等常用位图格式；
  - **Excel**：如 .xlsx、.xls；
  - **PDF**：.pdf。
- 支持 **多文件**（若现网已支持多选则保留；否则在方案中约定单文件先行再迭代），已选文件在输入区附近以 **列表或缩略图 + 文件名** 展示，并可 **删除** 单项。

**6.5.3 发送与识别中态**

- 用户点击 **发送**，将 **当前文本 + 已选文件** 一并提交 **识别/解析**（允许仅文、仅附件、混合），**不**自动建单。
- **识别中**期间（与 §6.4.5 对齐）：
  - **输入框**：保持 **可编辑**；
  - **文件按钮**、**发送按钮**：**禁用**（置灰或不可点），直至 **本轮识别结束**。
- 识别结束后恢复可操作；用户可继续增删附件、修改文字并再次发送。

**6.5.4 与移动端差异**

- PC 以 **本地文件** 方式接入 Excel/PDF 与图片；移动端以 **相机/相册** 为主接入图片。服务端 **识别接口** 统一抽象为「文本 + 附件列表」（见 §7.2）。
- **交付物（PC 增补）**：单附件 **大小上限**、**总个数上限**、不支持格式的 **前端拦截提示**、大 PDF/Excel 超时与拆分策略。

### 6.6 「吐槽」（替换点赞/点踩）

用于对 **本轮智能识别结果/体验** 提交文字反馈；**不落业务运单数据**，仅写入 **吐槽反馈记录** 供后台统计（见 **§7.5**）。

**6.6.1 入口与位置**

- **Web / 移动端** 一致：在 **本条识别结果全部运单组卡片之后**、与 **「确定建单（N单）」** 同属 **识别结果区底部操作带**（左右顺序由 UX 定稿，须 **同时可见、不误触**）。
- **不再**展示 **点赞、点踩** 或等价二元评价控件。

**6.6.2 交互**

- 点击 **「吐槽」** 打开 **弹窗**（或底部抽屉，移动端）：标题建议 **「吐槽」** 或 **「意见反馈」**。
- 正文为 **多行文本输入**；**最多 500 个字符**（按产品统一计数规则，如 Unicode 字符长度）；输入区展示 **已输入 / 500**；超出部分 **禁止继续输入** 或 **截断并提示**（与现网输入规范一致）。
- **取消**：关闭弹窗，不提交。
- **提交**：将 **吐槽正文** 发往服务端落库（**§7.5**）。**建议**：正文 **去空白后至少 1 个字符** 才可提交，避免无意义空记录；提交中 **防重复点击**。
- **提交成功**：轻提示（如 Toast **「已提交，感谢反馈」**）并关闭弹窗；**不强制**跳转、**不影响**已展示的识别结果与建单勾选状态。同一用户 **可多次** 对多轮识别分别吐槽。

**6.6.3 与建单的关系**

- 吐槽 **独立于** 建单：未点「确定建单」也可提交吐槽；**仅吐槽不建单** 允许。

## 7. 数据与接口（产品级）

### 7.1 客户简称扩展配置（建议模型）

| 字段 | 说明 |
|------|------|
| id | 主键 |
| tenant_id | 租户 |
| shortcut | 客户简称 / 触发词（唯一性或联合唯一由实现定；实现层可沿用字段名 `code`） |
| expansion | 展开说明（长文本，可约定为「键值对」或 JSON 由实现解析；实现层可沿用 `explanation`） |
| sort_order | 匹配优先级 |
| enabled | 是否启用 |
| updated_by / updated_at | 审计 |

### 7.2 识别接口（逻辑）

- 入参：**原始文本**（可空）、**附件列表**（可空）：每项含 **类型**（如 `image` / `excel` / `pdf`）与 **文件内容或上传后 URL**；移动端对应相册/相机图片，PC 对应用户选择的图片/Excel/PDF。`tenant_id`、可选 `user_id`；**文本与附件至少一项非空**。服务端可编排：**图片/PDF** 走 OCR 或视觉多模态、**Excel** 走表格解析、再与文本一并送入 **doubao-seed-2-0-pro** 或规则层（以技术方案为准）。
- 出参：**逻辑上**为运单组列表。若采用 **§10.3** 嵌套形态：每组含 **`group_key`（装/卸/货，模型侧聚类键，≠ 前端一张卡片）**、`items[]`（每条含车牌、司机、`trip_count`、字段键值、`build_count` 等）、`applied_shortcuts[]`（本次命中的客户简称；若沿用旧接口可为 `applied_codes[]`，语义相同）。**下发助手前**须映射为 **五元组卡片** 列表（见同条下一项）。若识别服务统一走大模型 JSON，**逻辑契约与示例见 §10.3**。
- **与助手 UI 对齐**：前端展示的 **一张运单组卡片** = **五元组**（装、卸、车线、车牌、司机）唯一的一行；若模型仍输出「`group_key` + 多条 `items`」结构，**服务端须展开/拆分为多张卡片**（不同 `plate` 或不同 `driver` 不得挤在同一张卡片内），每张卡片带独立 **`build_count`（建单数量）**。详见 **§1.4、§4.1**。
- **返回前处理**：出参须经 **§3.1 主数据补全** 流水线后再下发前端；前端展示以补全结果为准。
- 错误码：文本过长、无有效字段、服务超时等。

### 7.3 批量建单与进度（逻辑）

- **提交**：根据用户选中的运单组及每组 `build_count`，生成 **P** 条创建运单请求（或等价批量接口）；返回 **任务 ID** 或流式进度通道。
- **进度**：客户端轮询/推送获取 **成功数、失败数、完成数**；用于驱动「执行进度」展示。**助手侧** 以 **单条气泡** 承载「进行中 → 结束态」，见 **§3 步骤 9～10、§6.3.2～6.3.3**。
- **结果明细**：每组返回 **成功 / 失败** 及 **错误信息**（供侧栏与 **卡片就地编辑** 展示）；失败组允许携带 **字段级错误** 以便高亮（若现网支持）。

### 7.4 主数据匹配服务（逻辑）

- **触发时机**：识别（模型+规则）产出字段候选后、封装 API 响应前。
- **输入**：租户上下文 + 每条 `items[]` 上的待匹配字段（车牌、司机姓名/电话、装货地址、卸货地址等）。
- **输出**：写回 **标准展示字段**（及若开单接口需要，附带 **主数据 ID** 等，由研发与现网创建运单契约对齐）。
- **数据源**：**车辆信息管理**、**司机信息管理**、**地址库**、**车线管理**；装/卸地址匹配顺序为 **地址库优先，再车线管理**（与 §3.1 一致）。
- **多命中策略**：除产品另有说明外，**多条命中时取「最新」一条**（时间字段定义见 §3.1.1 表内脚注）；**零命中**时返回 **未匹配标记** + **§3.1.4 提醒文案** 所需字段，前端与运单组卡片联动展示。
- **边界**：不替代 **创建运单接口** 内部的关联与校验；仅保证助手侧 **回显与编辑初值** 与主数据一致、减少用户再录入。**自动新增车辆/司机** 由 **现网建单链路** 实现与否决定（见 §3.1.3）。

### 7.5 吐槽反馈记录（提交与落库）

**接口（逻辑）**

- **触发**：用户在 **§6.6** 弹窗内点击 **提交**。
- **入参（产品必填语义）**：
  - **`feedback_text`**：用户输入的吐槽正文，**长度 ≤ 500**（服务端须校验）；
  - **`group_corp_id`**（或现网等价字段名）：**提交人所属集团 ID**（由登录态/网关解析，**禁止**由前端伪造）；
  - **`user_name`**：**用户名**（展示名或登录名，与后台统计口径一致即可）；
  - **`user_mobile`**：**用户手机号**（登录态关联的手机号；若账号无手机则以空或占位策略由研发定，**须在方案中说明**）。
- **建议附加**（便于统计与排障，由研发与 §6.6 对齐）：`tenant_id`、提交时间、`session_id` / 识别批次号、客户端类型（Web/App）、可选 `prompt_version`（若与本轮识别绑定）。

**落库与统计**

- 服务端 **持久化** 上述记录；**租户/数据权限隔离** 与财运通规范一致。
- 后台须支持 **按时间、集团、用户** 等维度 **汇总/导出/查询**（具体报表或运营看板由数据/运营需求单独立迭代，本产品验收 **以「可查到单条记录且字段齐全」** 为准）。

## 9. 验收要点（摘要）

- 粘贴给定样例（含多组、多车牌、×2/*3 趟次）时，**条数（各组建单数量之和）、组数（五元组卡片数）、右上角建单数量**与 **§4** 一致。
- 五条必显字段每条运单均出现；运费、货值等存在时格式为 **两位小数**。
- 客户简称命中后，展开说明正确合并且无理由覆盖原文显式信息。
- **确定建单**：原识别区底部 **「确定建单（N单）」** 中 **N** 与 **已勾选且未锁定且未迁出** 组的 **建单数量合计** 一致；**迁入区「确定建单」** 仅针对当前迁入组，与本批实际上送物理运单数一致；**已锁定成功组不得计入 N、不得上送**；进行中 **不可重复误触** 导致重复建单（或具备幂等策略）。
- **部分成功 / 锁定**：首轮部分成功后，成功组 **无勾选与编辑**；失败组可单独或合并勾选再次提交；**点击修改** 失败气泡后 **仅勾选对应组** 的行为符合 **§6.3.3**；**全部组成功后** 底部 **确定建单** 隐藏。
- **进度与文案**：**单条建单气泡** 内 **组数 K、总单数 P** 与解析结果一致；执行进度 **已完成/总数** 与后台结果一致直至结束；**无** 「好的，财财收到」寒暄；结束后 **同气泡** 展示各组成功/失败块。
- **失败重提**：失败气泡 **「点击修改」** → 该组 **迁入失败气泡下方** + **仅勾选该组** + **就地编辑** + **迁入区确定建单**（**§6.2.1 / §3.10**）；原识别区 **确定建单** 不计已迁出组；允许 **未完成「完成」即建单**；可对其余失败组重复 **点击修改** 分别迁入或仍在原区勾选合并提交（以产品设计二选一或并存为准，PT 演示为 **迁入 + 本区建单**）。**已锁定成功组**（含 **仅会话气泡未更新、用户误点较早失败链**）点击 **「点击修改」** 时 **Toast**：**「运单组已建单完成，不能再修改。」**（与文首 **`reference/智能录单-历史失败气泡-已建单成功误点点击修改.png`** 场景一致）。
- **识别结果交互**：**司机姓名 / 主驾手机分列**；**仅** **「编辑 >」/「完成」就地编辑**（**§6.2**）；**无** 运单组编辑弹窗；**确定建单** 主入口在 **识别结果区最底部**，**失败迁入** 时在对应气泡下提供 **第二处确定建单**（**§6.3.1**）；与 [PT/index.html](PT/index.html) 线框一致可验收。
- **就地编辑（§6.2）**：卡片表格内切换只读/输入；**主数据未命中 / 建单失败** 等 **摘要提示条**；接口可定位字段 **标签+输入框红色**；**完成** 写回当前组、**不**在卡片内建单；与 [PT/index.html](PT/index.html) **§6.2.3** 演示一致可对照验收。
- Web 侧栏、PC **「识别配置」**（§5.4：简称 Tab + 映射 Tab、顶部说明、智能映射过滤 **12345测试** 类噪声不自动入库）流程可用（与平台 F004/F005/F010 行为一致）。
- **PC Web（财财）**：底部输入区为 **文件** 入口（非单独图片图标）；可选 PNG/JPG/JPEG、Excel、PDF；**发送** 提交识别；**识别中** 禁用 **文件**与 **发送**、输入框可编辑；行为与 §6.5 一致。
- **移动端**：空白入口、拍照/相册多图、缩略图增删、文本与提交显隐规则、**识别中** 禁用相机与提交且输入框可编辑、识别结果 **不自动建单**、**确定建单** 后才调创建运单接口；行为与 §6.4 一致。
- **模型与结构化输出**：在默认模型 **doubao-seed-2-0-pro**、约定 **prompt_version** 下，识别服务返回 JSON 符合 **§10.3** 骨架，`meta.physical_waybill_count` / `meta.group_count` 经服务端映射后与 **§4 五元组分组、建单数量** 一致，且可追溯 Prompt 版本。
- **主数据补全与回显**：片段车牌（如 `00123`）经 **车辆信息管理** 模糊匹配后，助手内展示 **完整标准车牌**（如 `京A00123`）；**多条命中**时展示与 **§3.1** 一致的「最新」策略；装/卸地址经 **地址库→车线管理** 后回显标准表述；**确定建单** 后仍走 **现网创建运单** 关联逻辑，助手不重复实现带出规则。
- **未命中提醒**：车辆/司机 **0 条** 命中时须出现 **「未查询到车辆」「未查询到司机」** 类提示（§3.1.4）；用户 **补全信息** 后建单，**自动新增档案** 行为与现网能力对齐并验收。
- **吐槽（§6.6 / §7.5）**：结果区 **无点赞/点踩**；**「吐槽」** 弹窗 **500 字** 上限与字数展示正确；提交成功后后台记录含 **集团 ID、用户名、用户手机号、吐槽正文**（及建议附加字段若实现）；**统计/查询** 可用。

### 10.1 默认模型与配置归属

| 项 | 约定 |
|----|------|
| V1.0 默认对话模型 | **doubao-seed-2-0-pro**（与火山方舟 / 公司内部模型网关实际产品名保持一致；若注册名有前缀后缀，以配置中心为准） |
| 可切换性 | 允许通过配置切换同系列或其它已对接模型；**验收与回归**需声明「当前默认模型 + Prompt 版本」 |
| 本文档范围 | **不承载**完整 System/User Prompt 正文，避免与研发配置库双源不一致 |
| 研发维护物 | 建议单独维护《智能录单 · 文本识别 Prompt 说明》或代码仓内 `prompts/`（含版本号、变更人、生效日期），本 PRD 引用其**产品约束**与**输出 Schema** |

### 10.2 Prompt 须覆盖的产品规则（摘要）

调用模型时，Prompt（或「规则 + Prompt」组合）须显式体现以下内容，并与 **§3～§5、§4.3** 一致：

1. **角色**：财运通运单文本解析器，仅基于用户输入与（如有）服务端注入的「客户简称扩展」展开结果做抽取，**不编造**车牌、电话、金额等；无法确定则留空或 `null`，不得臆测。**车牌等允许输出用户原文片段**（如 `00123`），**标准号牌展示由 §3.1 主数据补全完成**，模型不必猜测完整车牌。
2. **输入**：用户 **原始文本**（可空）与 **附件**（可空）：含 **图片**（多图）、**Excel**、**PDF** 等经网关解析后的结构化片段或摘要；若网关已做简称扩展合并，可同时传入 `expanded_context`（可选），模型以**原文显式字段优先**，扩展仅补空缺（同 §5.2）。**至少文本或附件其一非空**（与 §7.2 一致）。
3. **拆合单（与助手对齐）**：模型可先按装/卸/货聚类，再输出 `items[]`；**服务端**须将结果映射为 **五元组**（装、卸、车线、车牌、司机）颗粒度的多张卡片（见 **§1.4、§4.1、§7.2**）。每条助手卡片对应 **`build_count`（建单数量）**（默认 1，同一五元组多物理单用 `build_count`>1 表达）。输出 **物理运单总数** 与 **组数（卡片数）** 与 §4 一致。
4. **必显五字段**：每条运单条目必须包含 **`estimated_departure`**（预计发车时间）、**`route_name`**（车线名称，默认「装→卸」）、**`loading_place`**、**`unloading_place`**、**`plate_no`**，允许为空字符串，键不可省略。
5. **可选字段**：仅当文本中存在可靠依据时填充（主驾/副驾司机及手机号、押运员电话、挂车号、装车时间、运单号、货名、货值、重量、件数、应收运费等）；金额类保留两位小数语义（由后处理格式化为 `XXXX.00` 亦可）。助手 **卡片/就地编辑** 展示的可选列以 **§6.1** 现网裁剪为准；模型仍应输出完整键值供开单与扩展列使用。
6. **输出形态**：**仅输出合法 JSON**（或约定由网关剥离模型输出外层的 Markdown 代码块标记），无多余寒暄；便于服务端校验与落库。

### 10.3 识别结果 JSON 骨架（产品契约）

以下为**逻辑结构示例**，字段名可由研发与现网接口对齐（中英文或 snake_case 统一一种）；**`meta` 与 `groups` 为必选顶层键**。

**与助手 UI 的对应关系**：`groups[].group_key` + `items[]` 表示 **模型/解析服务** 便于输出的 **嵌套结构**；**助手前端** 须收到 **展开后的扁平「卡片」列表**（每张卡片 = **五元组** + 业务字段 + **`build_count`**），与 **§1.4、§4.1、§7.2** 及 [PT/index.html](PT/index.html) 一致。若网关/API 直接返回扁平列表，字段语义与下例等价即可。**`meta.group_count` / `meta.physical_waybill_count` 的验收口径**以 **映射后** 的卡片数、各卡 `build_count` 之和为准，**不得**将 `meta.group_count` 简单等同于未展开前的 `groups.length`。

```json
{
  "meta": {
    "model": "doubao-seed-2-0-pro",
    "prompt_version": "string",
    "physical_waybill_count": 0,
    "group_count": 0,
    "summary_text": "本次共解析出 X 条运单，分为 Y 个运单组"
  },
  "applied_shortcuts": [
    { "shortcut": "string", "expansion_used": true }
  ],
  "groups": [
    {
      "group_id": "string",
      "group_key": {
        "loading_place": "string",
        "unloading_place": "string",
        "cargo": { "name": "string", "attrs": {} }
      },
      "items": [
        {
          "item_id": "string",
          "plate_no": "string",
          "trip_count": 1,
          "build_count": 1,
          "driver_name": "string",
          "driver_phone": "string",
          "trailer_no": "string",
          "estimated_departure": "string",
          "route_name": "string",
          "loading_place": "string",
          "unloading_place": "string",
          "cargo_name": "string",
          "cargo_value": null,
          "weight": null,
          "piece_count": null,
          "freight_receivable": null,
          "extra": {}
        }
      ]
    }
  ]
}
```

**说明**：

- `meta.physical_waybill_count`：等于 **映射为扁平卡片后** 各卡 **`build_count` 之和**（若 JSON 仍为嵌套结构，则等于展开后全部 `items` 行上 `build_count` 之和），与助手气泡中 **P**、执行进度总数一致。
- `meta.group_count`：**Y** = **五元组卡片数**（映射后与 **§1.4** 一致）；嵌套 `groups` 场景下须按 **§7.2** 拆分后再计数，不得仅用顶层 `groups` 个数。
- `trip_count` 与 `build_count`：若产品统一为「建单倍数」，可只保留 `build_count`，但须在接口文档中写清。
- `extra`：承载暂未进主表的键值，避免模型随意增删顶层键导致解析失败。

### 10.4 验收与回归

- 提供 **固定文本用例集**（含多组、多车牌、×2/*3、客户简称命中、金额、冲突场景），同一用例在 **同一模型 + 同一 `prompt_version`** 下输出 JSON 须通过 Schema 校验且 **meta 计数**与人工标注一致。
- 更换模型或升级 Prompt 时：**升级 `prompt_version`**，并跑全量用例回归。

### 10.5 与通用识别平台的关系

- **2604-智能识别平台-PRD-v1.0.md** 中若已约定大模型品牌与网关，本需求 **默认走同一网关**，模型名配置为 **doubao-seed-2-0-pro**；冲突时以**配置中心生效值**为准，并回写本文档版本记录。
