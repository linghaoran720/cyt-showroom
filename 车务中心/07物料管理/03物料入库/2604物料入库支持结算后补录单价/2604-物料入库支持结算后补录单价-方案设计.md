## 1. 问题背景

### 1.1 客户问题

客户在物料采购入库时无法指定物料的具体单价，采购单价采用阶梯定价模式（用量越多，单价越低）。需要在一定周期后结算时，根据采购的用量情况计算出具体的价格。

**核心痛点**：
- 入库时无法确定单价，但系统要求必填
- 只能填写预估价格，与实际价格存在差异
- 结算后需要手动调整入库单价和相关费用
- 已发放的物料费用无法自动更新

### 1.2 系统现状

**当前入库流程**：
1. 仓管员登记入库，必须填写采购单价（必填字段）
2. 系统计算采购总价 = 采购单价 × 入库数量
3. 物料发放时使用入库单价计算发放费用
4. 结算时发现实际价格与预估价格不一致
5. 需要手动调整入库单和发放单的费用

**存在的问题**：
- 采购单价必填，无法适应阶梯定价场景
- 手动调整费用工作量大且容易出错
- 已发放物料费用无法批量更新
- 成本核算不准确

### 1.3 业务场景

**典型场景**：
```
时间线：
Day 1: 入库100升机油，单价未知，暂不填写
Day 5: 发放30升给车队A
Day 10: 发放40升给车队B
Day 30: 结算时确定总价5000元，单价50元/升
       需要更新：
       - 入库单：单价50元/升，总价5000元
       - 车队A发放：单价50元/升，总价1500元
       - 车队B发放：单价50元/升，总价2000元
```

## 2. 解决方案

### 2.1 方案对比

经过分析，我们选择了**方案 D：灵活补录 + 智能聚合**

| 方案 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| 方案 A：单独补录功能 | 功能独立，不影响现有流程 | 开发工作量大 | ⭐⭐⭐ |
| 方案 B：详情页补录 | 操作便捷，工作量小 | 不支持批量，需手动触发 | ⭐⭐ |
| 方案 C：智能补录 + 分阶段更新 | 功能完整，数据安全 | 流程稍长，工作量大 | ⭐⭐⭐⭐ |
| **方案 D：灵活补录 + 智能聚合** | **功能完整，用户体验好，扩展性强** | **开发工作量较大** | **⭐⭐⭐⭐⭐** |

### 2.2 选定方案

**方案 D：灵活补录 + 智能聚合**

**选择理由**：
1. **功能完整**：支持单个和批量补录，满足所有业务场景
2. **用户体验好**：物料聚合显示，输入联动，批量校验提示
3. **数据准确**：自动计算和分配价格，自动更新所有关联数据
4. **扩展性强**：后续可增加价格历史对比、价格变动分析等功能
5. **技术可行**：聚合算法简单高效，价格分配逻辑清晰

## 3. 方案设计

### 3.1 功能模块

#### 模块 1：入库登记优化

**功能说明**：
- 采购单价字段改为选填（当前为必填）
- 单价为空时，采购总价也为空
- 保存时不校验单价必填

**影响范围**：
- 前端：入库登记表单
- 后端：入库保存接口校验规则

#### 模块 2：补录价格功能

**功能说明**：
- 列表页新增"补录价格"按钮
- 支持单选和多选入库单
- 点击后进行条件校验
- 打开补录价格弹窗

**补录条件**：
- **条件 A**：总价 & 单价都为空 → 可补录
- **条件 B**：总价 & 单价不为空 且 未发放 → 可补录
- **限制条件**：
  - 已报销 → 不可补录
  - 已发放（且不满足条件B）→ 不可补录

**批量校验**：
- 逐个校验每个入库单是否满足条件
- 不满足的显示原因列表
- 提示："3个入库单可补录，2个不满足条件，是否继续？"
- 用户可选择：继续补录 / 取消

#### 模块 3：物料聚合显示

**功能说明**：
- 按物料编码聚合显示
- 显示物料编码、物料名称、总数量
- 用户可选择需要补录的物料（支持部分补录）

**聚合规则**：
```
示例：
入库单1：机油M001 30升
入库单2：机油M001 40升
入库单3：机油M001 30升

聚合后：
机油M001 总计100升（来自3个入库单）
```

#### 模块 4：价格输入与计算

**功能说明**：
- 支持输入单价，自动计算总价
- 支持输入总价，自动计算单价
- 实时联动计算

**计算规则**：
- 输入单价 → 总价 = 单价 × 总数量
- 输入总价 → 单价 = 总价 ÷ 总数量
- 单价保留2位小数，四舍五入

#### 模块 5：附件上传

**功能说明**：
- 补录价格时可上传账单附件
- 支持上传图片（JPG、PNG）和PDF
- 附件同步到每个入库单中

**附件处理**：
- 上传的附件保存到文件服务器
- 附件URL记录到入库单的附件字段
- 多个入库单共享同一批附件

#### 模块 6：价格分配与更新

**功能说明**：
- 将聚合后的单价分配到各入库单
- 更新入库单的单价和总价
- 更新库存表的单价和总价
- 更新发放记录的单价和总价

**分配规则**：
```
场景：物料M001在3个入库单中
- 入库单1：30升
- 入库单2：40升
- 入库单3：30升
- 总计：100升，补录单价50元/升

分配方式：
- 入库单1：单价50元/升，总价1500元
- 入库单2：单价50元/升，总价2000元
- 入库单3：单价50元/升，总价1500元
```

#### 模块 7：补录状态标识

**功能说明**：
- 入库单增加"是否已补录价格"字段
- 列表页显示补录状态（已补录/未补录）
- 支持按补录状态筛选

#### 模块 8：补录记录追溯

**功能说明**：
- 记录补录操作人和补录时间
- 记录补录前后的价格变化
- 支持查看补录历史

### 3.2 业务流程

```
1. 仓管员点击【登记】按钮
   ↓
2. 填写入库信息（网点、入库类型、物料、数量）
   ↓
3. 采购单价留空（不填）
   ↓
4. 点击【确定】保存
   ↓
5. 系统生成入库单号，单价和总价为空
```

#### 流程 2：物料发放（使用物料维护表单价）

```
1. 调度员提交物料申请
   ↓
2. 审核通过后发放物料
   ↓
3. 系统使用物料维护表的单价计算发放费用
   ↓
4. 生成发放记录
```

#### 流程 3：补录价格（单个入库单）

```
1. 财务人员勾选1个入库单
   ↓
2. 点击【补录价格】按钮
   ↓
3. 系统校验是否可补录
   ↓
4. 打开补录价格弹窗
   ↓
5. 显示物料明细（按物料编码聚合）
   ↓
6. 用户勾选需要补录的物料
   ↓
7. 输入单价或总价（自动联动计算）
   ↓
8. 上传账单附件（选填）
   ↓
9. 点击【确定补录】
   ↓
10. 系统更新入库单、库存、发放记录
    ↓
11. 附件同步到入库单
    ↓
12. 记录补录日志
```

#### 流程 4：补录价格（批量入库单）

   ↓

   ↓

   ↓

   - 3个可补录

   ↓

   ↓
6. 打开补录价格弹窗
   ↓
7. 显示物料明细（按物料编码聚合，来自多个入库单）
   ↓
8. 用户勾选需要补录的物料
   ↓
9. 输入单价或总价（自动联动计算）
   ↓
10. 上传账单附件（选填）
    ↓
11. 点击【确定补录】
    ↓
12. 系统按物料编码分配价格到各入库单
    ↓
13. 批量更新入库单、库存、发放记录
    ↓

### 3.5 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 补录时入库单已报销 | 提示"入库单已报销，不可补录"，不允许操作 |
| 补录时入库单已发放（不满足条件B）| 提示"入库单已发放，不可补录"，不允许操作 |
| 批量补录时部分不满足条件 | 显示不满足条件的入库单和原因，用户可选择继续补录满足条件的 |
| 补录时输入的总价为0或负数 | 提示"总价必须大于0"，不允许保存 |
| 补录时附件上传失败 | 提示"附件上传失败，请重试"，不影响价格补录 |
| 补录时更新发放费用失败 | 回滚所有操作，提示"补录失败，请重试" |

### 4.5 核心算法

#### 算法 1：物料聚合

```javascript
/**
 * 按物料编码聚合入库单中的物料
 * @param {Array} inboundBills - 入库单列表
 * @returns {Object} 聚合后的物料数据
 */
function aggregateMaterials(inboundBills) {
  const aggregated = {};

  inboundBills.forEach(bill => {
    bill.materials.forEach(material => {
      const code = material.code;

      if (!aggregated[code]) {
        aggregated[code] = {
          code: code,
          name: material.name,
          unit: material.unit,
          totalQuantity: 0,
          bills: []
        };
      }

      aggregated[code].totalQuantity += material.quantity;
      aggregated[code].bills.push({
        billId: bill.id,
        billCode: bill.code,
        quantity: material.quantity
      });
    });
  });

  return Object.values(aggregated);
}
```

#### 算法 2：价格分配

```javascript
/**
 * 将聚合后的单价分配到各入库单
 * @param {String} materialCode - 物料编码
 * @param {Number} unitPrice - 单价
 * @param {Array} bills - 入库单列表
 */
function distributePriceToInboundBills(materialCode, unitPrice, bills) {
  const updates = [];

  bills.forEach(bill => {
    const totalPrice = (unitPrice * bill.quantity).toFixed(2);

    updates.push({
      billId: bill.billId,
      materialCode: materialCode,
      unitPrice: unitPrice,
      totalPrice: parseFloat(totalPrice)
    });
  });

  return updates;
}
```

#### 算法 3：批量校验

```javascript
/**
 * 批量校验入库单是否可补录
 * @param {Array} billIds - 入库单ID列表
 * @returns {Object} 校验结果
 */
function validateInboundBills(billIds) {
  const results = {
    valid: [],
    invalid: []
  };

  billIds.forEach(billId => {
    const bill = getInboundBill(billId);
    const reason = checkCanFillPrice(bill);

    if (reason === null) {
      results.valid.push(billId);
    } else {
      results.invalid.push({
        billId: billId,
        billCode: bill.code,
        reason: reason
      });
    }
  });

  return results;
}

/**
 * 检查入库单是否可补录价格
 * @param {Object} bill - 入库单对象
 * @returns {String|null} 不可补录的原因，null表示可补录
 */
function checkCanFillPrice(bill) {
  // 已报销不可补录
  if (bill.isExpensed) {
    return "已报销，不可补录";
  }

  // 条件A：总价 & 单价都为空
  if (!bill.unitPrice && !bill.totalPrice) {
    return null; // 可补录
  }

  // 条件B：总价 & 单价不为空 且 未发放
  if (bill.unitPrice && bill.totalPrice && !bill.isReleased) {
    return null; // 可补录
  }

  // 已发放不可补录
  if (bill.isReleased) {
    return "已发放，不可补录";
  }

  return "不满足补录条件";
}
```

### 5.3 风险评估

| 风险项 | 影响程度 | 应对策略 |
|--------|---------|---------|
| 已发放物料数量与入库数量不一致 | 高 | 按比例分摊费用，确保总费用一致 |
| 补录价格后报销单已生成 | 高 | 已报销的入库单不允许补录价格 |
| 补录操作失败导致数据不一致 | 高 | 使用事务确保数据一致性，失败时回滚 |
| 批量补录时性能问题 | 中 | 超过100条记录时使用异步处理 |
| 附件上传失败 | 中 | 附件上传失败不影响价格补录，可后续补传 |
| 历史数据兼容性 | 低 | 历史数据保持不变，新功能仅对新数据生效 |
