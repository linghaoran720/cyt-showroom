## 1. 概述

### 1.1 项目背景

物料盘点功能（v1.0）已上线，当前流程为：仓管员创建盘点单 → 录入盘点数量 → 提交后直接调整库存。在实际运营中发现以下问题：

- 登记员（仓管员）录入盘点数量时存在错误，提交后库存立即变更
- 发现错误后频繁执行「撤销 → 修改 → 重新提交」操作，导致库存数据频繁波动
- 缺乏审核环节，无法在库存变更前拦截错误数据，也难以追溯责任

客户要求在盘点提交与库存生效之间增加审批环节，由审批人确认盘点数据无误后才执行库存调整。

### 1.2 项目目标

- 盘点单提交后进入审批流程，支持多级审批，所有层级通过后才执行库存调整
- 支持取消审批，仓管员可在审批流转前撤回
- 审批驳回后仓管员可修正数据，保存后恢复原审批流程继续审批
- 对接独立审批模块，本功能不涉及审批流程配置

### 1.3 项目范围

**包含（In Scope）**：
- 盘点单提交后发起审批，状态变为「待审批」
- 支持多级审批，审批流转中状态为「审批中」
- 所有审批层级通过后执行库存调整（逻辑与原提交一致）
- 审批驳回，记录驳回原因，提交人修改保存后恢复原审批流程
- 取消审批，盘点单回到草稿状态
- 盘点单列表和详情页适配新状态（含审批中）
- 审批操作日志
- 审批通过时库存二次校验及异常处理

**不包含（Out of Scope）**：
- 审批流程配置（审批人、审批层级、审批条件）— 由审批模块管理
- 审批模块本身的开发
- 移动端审批
- 批量审批

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 审批模块 | 独立的通用审批系统，负责审批流程配置、审批人指派、审批流转等 |
| 多级审批 | 审批流程包含多个审批层级，需逐级通过后才最终生效 |
| 发起审批 | 盘点单提交时调用审批模块接口，创建一条审批任务 |
| 取消审批 | 提交人在审批流转前主动撤回审批请求 |
| 恢复审批 | 驳回后提交人修改保存，审批流程回到驳回前的审批节点继续流转 |
| 审批回调 | 审批模块处理完成后，通过回调接口通知盘点功能审批结果 |
| 审批异常 | 终审通过时因库存不足无法执行调整的异常状态 |

## 3. 需求详述

### 3.1 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| F001 | 提交发起审批 | P0 | 提交盘点单时发起审批，状态变为「待审批」，不直接调整库存 |
| F002 | 审批通过执行库存调整 | P0 | 支持多级审批，所有层级通过后二次校验库存并执行调整 |
| F003 | 审批驳回 | P0 | 接收审批驳回回调，记录驳回原因，提交人修改保存后恢复原审批流程 |
| F004 | 取消审批 | P0 | 提交人主动取消待审批的盘点单（仅限审批流转前） |
| F005 | 列表及详情适配 | P1 | 状态筛选、状态标签、详情页审批信息展示（含审批中状态） |
| F006 | 审批操作日志 | P2 | 记录审批相关操作事件 |

### 3.2 功能详细说明

#### F001 提交发起审批

**变更说明**：修改原F003（提交盘点单）的提交行为，提交后不再直接执行库存调整。

**提交流程**：
1. 仓管员点击「提交」按钮
2. 执行库存校验（与原逻辑一致）：所有盘亏项当前系统库存 ≥ 减少数量，否则拦截
3. 校验通过后弹出提交确认弹窗，展示盘点汇总
4. 用户确认后：
   - 盘点单状态变为「待审批」
   - 调用审批模块「发起审批」接口，传入：盘点单ID、盘点单号、提交人、盘点网点、盘盈/盘亏项数、盘亏金额、盘盈金额
   - 页面提示"已提交审批，请等待审批结果"
5. **不执行**库存调整、入库单/出库单生成

**业务规则**：
- 提交时仍执行库存校验，提前拦截明显不合理的数据
- 审批业务类型标识为「物料盘点」
- 草稿、已撤销状态可提交，提交后统一进入「待审批」
- 审批模块开始流转后（第一级审批人开始处理），盘点单状态自动变为「审批中」

**交互说明**：
- 提交按钮位置不变
- 提交确认弹窗内容不变，增加提示文案"提交后将进入审批流程"
- 提交成功后的提示从"提交成功"改为"已提交审批，请等待审批结果"

---

#### F002 审批通过执行库存调整

**触发方式**：审批模块通过回调接口通知终审通过（所有审批层级均已通过）。

**执行流程**：
1. 接收审批模块回调，验证盘点单状态为「审批中」
2. 区分回调类型：
   - **单级通过**：当前层级审批通过，盘点单保持「审批中」状态，审批模块自动流转至下一级审批人
   - **终审通过**：所有层级均通过，继续执行以下步骤
3. 二次校验当前库存充足性（盘亏项当前库存 ≥ 减少数量）
4. 校验通过：
   - 盘点单状态变为「已通过」
   - 执行库存调整（与原提交逻辑完全一致）：
     - 更新物料当前库存和可用库存
     - 盘亏项按FIFO关联入库单扣减，生成出库记录
     - 盘盈项自动生成入库单（来源：盘点）
   - 记录操作日志
5. 校验不通过：
   - 盘点单状态变为「审批异常」
   - 记录异常原因（具体哪些物料库存不足）
   - 通知提交人处理（系统消息/站内信）

**业务规则**：
- 由于提交到终审通过可能存在较长时间差（多级审批），必须在执行库存调整前重新校验库存
- 审批异常的盘点单，提交人需撤销后重新盘点
- 库存调整在同一事务中完成，全部成功或全部回滚

---

#### F003 审批驳回

**触发方式**：审批模块通过回调接口通知某一级审批人驳回。

**执行流程**：
1. 接收审批模块回调，验证盘点单状态为「审批中」
2. 盘点单状态变为「已驳回」
3. 保存驳回原因（由审批模块传入）
4. 记录操作日志

**驳回后修改保存**：
- 已驳回的盘点单，提交人可直接修改盘点数据
- 编辑页面加载原有数据，系统库存刷新为当前最新值
- 修改保存后调用审批模块「恢复审批」接口，盘点单状态恢复为「审批中」
- 审批流程回到驳回前的审批节点继续流转（非重新发起审批）
- 已驳回的盘点单也可删除（放弃本次盘点）

**交互说明**：
- 盘点单详情页展示驳回原因，红色文字高亮
- 已驳回状态下操作按钮显示「修改」和「删除」
- 修改保存成功后提示"已保存并恢复审批流程"

**技术说明**：
- 驳回原因由审批模块传入，本功能负责存储和展示
- 恢复审批时需传入 approval_id，审批模块根据该ID定位到驳回前的审批节点
- 修改保存时需重新执行库存校验（盘亏项当前库存 ≥ 减少数量）

---

#### F004 取消审批

**入口**：盘点单列表操作列「取消审批」按钮 / 盘点详情页「取消审批」按钮

**执行流程**：
1. 仅「待审批」状态可执行，且仅限提交人本人操作
2. 弹出确认弹窗：提示"取消后盘点单将回到草稿状态，可重新编辑后提交"
3. 用户确认后调用审批模块「撤销审批」接口
4. 审批模块返回成功：盘点单状态变为「草稿」，可重新编辑和提交
5. 审批模块返回失败（审批已流转）：提示"审批已在处理中，无法取消"

**业务规则**：
- 取消审批仅限「待审批」状态（审批尚未开始流转）
- 若审批已开始流转（状态为「审批中」），取消审批不可用
- 取消审批不涉及库存操作（因为库存尚未变更）
- 取消后盘点单回到「草稿」而非「已撤销」，因为库存从未生效

---

#### F005 列表及详情适配

**盘点单列表变更**：

| 变更项 | 原设计 | 新设计 |

| 状态标签颜色 | 草稿-灰、已提交-蓝、已撤销-灰 | 草稿-灰、待审批-橙、审批中-蓝、已通过-绿、已驳回-红、已撤销-灰 |

**盘点单详情页变更**：
- 新增「审批信息」区域，位于基本信息下方，包含：
  - 审批状态
  - 当前审批节点/审批人（审批中时显示当前待审批的层级和审批人）
  - 审批进度（已通过N级/共M级）
  - 审批时间（审批通过/驳回时显示）
  - 驳回原因（驳回时显示，红色文字）
- 操作按钮根据状态动态显示：
  - 待审批：「取消审批」
  - 审批中：无操作按钮（仅查看审批进度）
  - 已通过：「撤销盘点」
  - 已驳回：「修改」「删除」
  - 草稿/已撤销：「编辑」「删除」「提交」

#### F006 审批操作日志

**变更说明**：在原F010操作日志基础上，增加审批相关事件类型。

**新增事件类型**：

| 事件类型 | 触发时机 | 记录内容 |
|----------|----------|----------|
| 提交审批 | 盘点单提交时 | 操作人、操作时间 |
| 取消审批 | 取消审批成功时 | 操作人、操作时间 |

### 3.3 状态流转

```
草稿 ──提交──→ 待审批 ──审批流转──→ 审批中 ──终审通过──→ 已通过（库存生效）──撤销──→ 已撤销
                                      │                        │
                                      ├──审批驳回──→ 已驳回     └── 审批异常（库存校验失败）
                                      │               │
                                      │               └──修改保存──→ 审批中（恢复原审批节点）
                                      │
                                      └──（单级通过）──→ 审批中（流转下一级）
                 │
                 └──取消审批──→ 草稿
```

| 状态 | 状态值 | 可执行操作 |
|------|--------|-----------|
| 草稿 | 0 | 编辑、删除、提交 |
| 待审批 | 1 | 取消审批（提交人） |
| 审批中 | 2 | 查看审批进度（提交人）、审批通过/驳回（当前级审批人） |
| 已通过 | 3 | 查看、撤销 |
| 已驳回 | 4 | 修改（保存后恢复审批）、删除 |
| 已撤销 | 5 | 编辑（重新提交）、删除 |
| 审批异常 | 6 | 查看、撤销（恢复后重新盘点） |

### 3.4 对原功能的影响

| 原功能 | 影响说明 |
|--------|----------|
| F003 提交盘点单 | 提交后不再直接执行库存调整，改为发起审批；库存调整移至终审通过后执行 |
| F004 撤销盘点单 | 撤销仅对「已通过」和「审批异常」状态生效；「待审批」状态使用「取消审批」替代 |
| F005 重新编辑 | 「已驳回」状态修改保存后恢复原审批流程（非重新发起）；「已撤销」状态重新提交则发起新审批 |
| F006 盘点单列表 | 状态筛选项变更，新增「审批中」状态，操作列按新状态调整 |
| 盘点单主表 status 字段 | 枚举值从 0/1/2 扩展为 0/1/2/3/4/5/6 |

### 3.5 非功能需求

- **数据一致性**：审批通过后的库存调整必须在事务中完成
- **幂等性**：审批回调接口需支持幂等，防止重复回调导致重复调整库存
- **超时处理**：审批模块接口调用超时时，提交操作应回滚，提示用户重试
- **并发控制**：审批通过时重新校验库存，防止审批期间库存被其他操作消耗

## 5. 数据需求

### 5.1 盘点单主表变更

| 字段名 | 类型 | 必填 | 变更说明 |
|--------|------|------|----------|
| status | tinyint | 是 | 枚举扩展：0草稿/1待审批/2审批中/3已通过/4已驳回/5已撤销/6审批异常（原：0草稿/1已提交/2已撤销） |
| approval_id | varchar(50) | 否 | 新增：审批模块返回的审批任务ID |
| approval_node | varchar(100) | 否 | 新增：当前审批节点标识（用于恢复审批时定位） |
| approval_progress | varchar(20) | 否 | 新增：审批进度，如"2/3"表示已通过2级共3级 |
| approval_user_id | bigint | 否 | 新增：最近一次审批操作人ID |
| approval_user_name | varchar(50) | 否 | 新增：最近一次审批操作人姓名 |
| approval_time | datetime | 否 | 新增：最近一次审批操作时间 |
| reject_reason | varchar(500) | 否 | 新增：驳回原因 |
| exception_reason | varchar(500) | 否 | 新增：审批异常原因 |

### 5.2 数据流转

```
提交 → 调用审批模块发起审批 → 保存 approval_id → status=1(待审批)
                                    ↓
审批模块回调（开始流转）→ status=2(审批中)
                                    ↓
审批模块回调（单级通过）→ 更新 approval_progress → status=2(审批中，流转下一级)
                                    ↓
审批模块回调（终审通过）→ 校验库存 → 执行库存调整 → status=3(已通过)
                                    ↓（校验失败）
                              status=6(审批异常), 记录 exception_reason
                                    ↓
审批模块回调（驳回）→ status=4(已驳回), 记录 reject_reason + approval_node
                                    ↓
提交人修改保存 → 调用审批模块恢复审批 → status=2(审批中，回到驳回节点)
                                    ↓
取消审批 → 调用审批模块撤销接口 → status=0(草稿), 清空 approval_id
```

## 6. 接口需求

### 6.1 对接审批模块接口

| 接口 | 方向 | 触发时机 | 主要参数 |
|------|------|----------|----------|
| 发起审批 | 盘点 → 审批模块 | 盘点单提交时 | 业务类型=物料盘点、业务ID=盘点单ID、业务单号=盘点单号、提交人、摘要信息 |
| 撤销审批 | 盘点 → 审批模块 | 取消审批时 | approval_id |
| 恢复审批 | 盘点 → 审批模块 | 驳回后修改保存时 | approval_id、业务ID=盘点单ID |
| 审批结果回调 | 审批模块 → 盘点 | 审批状态变更时 | approval_id、回调类型(流转/单级通过/终审通过/驳回)、审批人、审批进度、驳回原因 |

### 6.2 内部接口变更

| 接口 | 变更说明 |
|------|----------|
| 提交盘点单 | 移除库存调整逻辑，增加调用审批模块发起审批 |
| 新增：审批回调接口 | 接收审批结果，区分流转/单级通过/终审通过/驳回，执行对应逻辑 |

| 盘点单详情查询 | 返回审批相关字段（含审批进度、当前审批节点） |

## 7. 验收标准

### 7.1 功能验收

- [ ] 盘点单提交后状态变为「待审批」，不执行库存调整
- [ ] 提交时仍执行库存校验（盘亏项当前库存 ≥ 减少数量），校验不通过则拦截
- [ ] 提交成功后向审批模块发起审批请求
- [ ] 审批模块开始流转后盘点单状态自动变为「审批中」
- [ ] 多级审批场景下，单级通过后保持「审批中」状态并流转至下一级
- [ ] 所有审批层级通过后盘点单状态变为「已通过」，自动执行库存调整（逻辑与原提交一致）
- [ ] 终审通过时二次校验库存，不满足则标记「审批异常」并通知提交人
- [ ] 审批驳回后盘点单状态变为「已驳回」，详情页展示驳回原因
- [ ] 已驳回的盘点单可修改保存，保存后恢复原审批流程（回到驳回前审批节点继续）
- [ ] 待审批状态的盘点单，提交人可执行「取消审批」
- [ ] 取消审批成功后盘点单回到「草稿」状态
- [ ] 审批已流转（审批中）时取消审批不可用
- [ ] 盘点单列表状态筛选包含：草稿、待审批、审批中、已通过、已驳回、已撤销
- [ ] 各状态标签颜色正确：草稿-灰、待审批-橙、审批中-蓝、已通过-绿、已驳回-红、已撤销-灰
- [ ] 盘点详情页展示审批信息（审批状态、当前审批节点、审批进度、审批时间、驳回原因）
- [ ] 操作按钮根据状态正确显示
- [ ] 撤销仅对「已通过」状态生效（原逻辑不变）
- [ ] 审批相关操作记录到操作日志
- [ ] 审批回调接口支持幂等

### 7.2 兼容性验收

- [ ] 原有草稿状态盘点单功能不受影响
- [ ] 原有撤销逻辑不受影响（仅触发状态从「已提交」改为「已通过」）
- [ ] 历史已提交的盘点单数据迁移为「已通过」状态
