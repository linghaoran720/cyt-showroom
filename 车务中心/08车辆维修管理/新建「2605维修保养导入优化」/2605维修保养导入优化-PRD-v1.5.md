### 需求背景（与 PDF 一致）

客户场景：维修厂提供的维修单多为竖版/长明细。当前痛点：

1. 导入为横板模板，客户需复制粘贴适配模板才能导入。  
2. 单次最多 5 个项目，而客户一车一次维修常有 20+ 项。  
3. 维修编号不可重复；若编号可重复，客户可接受分批导入（先 5 条再 5 条等）。

## 一、产品概述

### 1.1 背景与问题

财运通已支持维修结算单 Excel 导入，但现有模板采用「项目一～项目五」等**固定槽位**，明细条数受限；且模板要求用户维护合计类字段，与录入页「**先填项目明细、由系统自动计算费用合计**」的逻辑不一致，导致客户大单无法导入或重复做工。

### 1.2 产品定义

在**不改变「从 0 新建模块」**的前提下，对**线上维修结算单导入能力**做迭代：改为**长表**（一行一条项目明细），以**维修编号**为唯一聚合键合并为多行明细的一张结算单；费用、默认值、付款行规则与线上一致并对齐本 PRD。**保养结算单导入**的增量字段见 **4.7**（与维修共用同一套底层能力）。

### 1.3 产品目标

- 解除「每单固定 N 条项目」带来的导入上限问题，支持同一维修编号下任意行数的项目明细（上限若存在，遵循现网技术约束，本需求不另设业务封顶除非现网已有）。
- 导入模板与录入页一致：**费用合计由系统根据明细汇总计算**，用户无需在 Excel 中填写单据级合计列（若模板仍保留历史列，导入时**忽略**或后续从模板移除，见字段表说明）。
- 降低分组歧义：**仅维修编号**参与聚单；表头冲突**不阻断**，以首行为准，由客户自行修正数据。
- 改善导入体验：由「仅事后报错文件」改为**中间表预览与行内修正**（参考现网「导入物料申请（车辆）」等能力），**提交前**可编辑；**正式落库**仅在用户点击「提交」后执行。

### 1.4 核心价值

| 维度 | 价值 |
|------|------|
| 客户侧 | 与客户纸质/对账单「一单多行明细」结构一致，大单可导；减少在表格外手工汇总。 |
| 产品一致性 | 与录入页「明细驱动费用」一致，避免「表算一遍、系统再算一遍」。 |
| 实施 | 规则简单（编号聚单、首行表头、付款逐行），便于开发与测试。 |
| 可纠错性 | 中间表支持行内修改与校验结果展示，减少反复导出 Excel、对报错行号的成本。 |
| 数据完整性 | **同维修编号原子提交**，避免部分落库导致单据缺明细或重复编号无法二次提交。 |

## 二、用户与场景

### 2.1 角色

| 角色 | 说明 |
|------|------|
| 车队财务 / 车务管理员 | 批量导入维修结算数据，核对生成单据。 |
| 系统管理员（可选） | 配置导入模板可见列（若现网支持模板配置）。 |

### 2.2 核心场景

1. 客户 Excel 中**同一维修编号**占多行，每行一条配件/工时类明细，导入后生成**一张**维修结算单，项目明细条数与行数一致。
2. 导入行携带**支付方、付款方式**可为空，保存时按**现网默认值**赋值（与手工录入一致）。
3. 每行对应**一条付款信息**；允许多行同为「现金」等，**不合并**为一条付款记录。
4. 维修编号在系统中**已存在**时，在**提交落库阶段**校验**拒绝**（不覆盖、不更新）。
5. 用户在中间表中对齐、修正数据后，**整批或按规则提交**；**同一维修编号下所有行均校验通过**后，该单才允许生成，**禁止部分行先落库**。
6. 用户点击**导入**展开中间表后，**可不传 Excel**，直接在表格中**手动新增行**并填写，与上传解析后的编辑、提交流程相同。

## 三、功能说明

### 3.1 功能清单

| 功能 | 描述 |
|------|------|
| 中间表导入交互 | 用户在页面点击**「导入」**后**展开**抽屉/弹窗，展示**可编辑中间表**；可在此**上传 Excel** 将内容解析进表，也可**不上传**、仅靠**手动加行**维护数据。**提交前**不写入维修结算单主数据；形态参考现网「导入物料申请（车辆）」（模板下载、选文件、表格预览、校验结果列、提交、行首 +/- 增删行）。 |
| 两阶段校验 | **阶段一（字段/格式）**：在中间表展示过程中可对每行做必填、格式等校验，结果展示在「校验结果」列（或通过行状态提示），用户可就地改单元格。**阶段二（提交落库）**：用户点击「提交」后执行**数据完整性/业务校验**（如维修编号在系统中是否已存在、字典是否存在等），再按规则生成结算单。 |
| 同编号原子提交 | 按**维修编号**分组：任一组内**只要存在一行未通过阶段一校验**，则该**整组**不可参与生成（提交按钮可整体禁用或点击后明确提示）；**不允许**仅将组内已通过行落库。**禁止部分提交**，避免生成「缺明细的单据」或再次提交时被判定**维修编号重复**而无法补全。 |
| 长表导入 | 一行一条项目明细；同一维修编号合并为一张结算单。 |
| 聚合规则 | 仅按维修编号相同判断；编号必填且非空。 |
| 表头取值 | 维修费用信息类字段在同一编号组内：以**组内文件行序第一行**为准；首行单元格为空则该表头字段为空，**不向后行顺延**；多行表头值不一致**不报错**。 |
| 配件费计算 | 配件费为空时按 **单价×数量** 写入；配件费非空则以导入值为准。 |
| 费用汇总 | 单据级配件费/工时费/其他费/费用合计由系统按明细汇总，与录入页逻辑一致。 |
| 三类费用校验 | **取消**「配件费、工时费、其他费不能同时为空」的导入专项校验；与**现网录入**一致，允许均为空。 |
| 支付默认值 | 支付方、付款方式**非必填**；为空时按现网默认（如公司付费、现金）赋值。 |
| 付款行 | **不聚合**；每行一条付款记录；允许重复付款方式。 |
| 备注合并 | 同一编号多行备注合并为一条，**英文逗号**分隔；超长按**现网该字段上限截断**。 |
| 重复编号 | 与系统已有维修结算单**维修编号**冲突：在**阶段二（提交落库）** **拒绝**该编号对应整单的生成（**语义为拒绝写入**）；中间表阶段可仅提示或待提交时再统一校验（以实现为准，但不得出现「半单已落库」）。 |
| 中间表数据带入 | 用户上传并解析 Excel 后，以**单元格内容原样**填入中间表（含空值、文本格式数字等）；**不在此阶段**执行业务聚合、费用重算、与系统主数据对账等**落库级逻辑**。用户可随时**手动新增/删除行**（与物料申请导入一致）。 |
| 上传覆盖中间表 | 每次上传并解析成功后，**覆盖当前中间表已有内容**（先清空再写入本次解析结果，或等价实现）；**不追加**到已有行后。用户手填未保存的数据被覆盖前需**二次确认**（文案与现网一致）。 |
| 中间表控件类型 | 各字段在中间表内**下拉模糊搜索**与**仅手动输入**约定见第四章；本版已按**登记维修结算记录**页与录入组件一致性预填。 |
| 会话与关闭提示 | 未提交关闭弹窗/页面导致中间表数据丢失的，需有**与现网物料申请导入一致**的提示或行为（见参考交互）。 |


### 3.2 导入交互说明（中间表，参考「导入物料申请（车辆）」）

**目标**：解决原流程「只能拿到导入结果文件再改 Excel」的痛点；用户可在系统内预览、改错后再落库。

**整体步骤（与参考能力对齐）**

1. **下载模板**：支持「模板设置」「下载模板」（若现网维修导入已具备则沿用）。
2. **打开中间表**：用户在业务入口点击**「导入」**，**展开**抽屉/弹窗，展示中间表区域（可与物料申请导入一致：左侧步骤说明 + 右侧表格）。**此时即可手动点「+」增行填写**，无需先上传文件。
3. **上传文件（可选）**：选择本地 Excel 后解析；文件大小、行数等限制**与现网一致或单独约定**。解析成功后，**覆盖当前中间表**（清空已有行再写入本次解析结果；若中间表已有未提交数据，覆盖前**确认提示**）。**上传解析成功后仍不落库**维修结算单。
4. **表格内容**：列为本需求字段（见第四章）及**校验结果**列；支持行内编辑；各字段控件类型（模糊下拉 / 纯手输）见第四章标记。
5. **阶段一校验**：对每行进行必填、格式等检查，不通过则在「校验结果」列展示原因（可多行错误合并展示）；单元格可标红或可定位；用户可直接修改表格单元格直至该行通过（具体 UI 与物料申请导入保持一致即可）。
6. **提交**：用户点击「提交」后进入**阶段二**：对**拟提交的数据**执行业务/完整性校验（如维修编号是否已在系统存在、组织/字典是否有效等），**全部内容通过后**按维修编号分组逐单生成结算单；若阶段二失败，**已通过的组也不应产生脏数据**（事务上整批回滚或仅失败组不落库，由研发实现，**不得**对同一维修编号只生成部分明细行）。
7. **同编号原子提交（强制）**  
   - 将所有行按 **维修编号** 分组。  
   - **仅当某一组内全部行均通过阶段一校验**，该组才允许进入阶段二生成。**生成后**，校验结果变为「**导入成功**」等终态时，**字段值不允许修改**。  
   - **禁止**：组内部分行校验失败时，仍将组内「已通过」行生成一张结算单——否则会出现**单据明细不完整**；用户再次导入修正行时，**维修编号已存在**导致**无法补全**或流程冲突。

### 3.3 业务流程（落库阶段简述）

在用户于中间表点击**提交**且阶段一、二均满足条件后：

1. 按 **维修编号** 分组；组内行序以当前中间表行序为准（与用户调整后的顺序一致）。
2. 每组生成一张维修结算单：表头字段取自该组**行序第一行**；项目明细为组内每行一条；付款信息为组内每行一条（不合并）。
3. 配件费按规则计算后，汇总费用合计；备注合并截断；支付默认值补全。
4. 阶段二失败时给出明确原因（行号、维修编号、失败类型）；**同一提交批次内**对已生成数据的事务性要求见3.2。

### 3.4 与旧模板差异（迁移说明）

| 项目 | 旧模板（现状） | 新模板（本需求后） |
|------|----------------|---------------------|
| 项目列 | 项目一～项目五等宽表槽位 | 长表：一行一条项目 |
| 合计列 | 可能存在单据级合计 | **不要求用户填写**；以系统汇总为准 |
| 聚合 | 隐式一单一行或槽位限制 | **显式维修编号**聚单 |
| 导入交互 | 上传后直接或异步生成结果 | **中间表预览 + 提交后落库**；两阶段校验；**上传覆盖**中间表 |

## 四、字段全量说明（导入与业务）

以下表格为需求及约定。列含义：

- **是否支持导入模板配置**：是否允许在「导入模板配置」中控制该列是否出现在可下载模板中（若现网无配置能力，则全部为「是」表示「应出现在标准模板列清单中」）。
- **是否默认进入导入模板**：新建/重置模板时，该列是否默认勾选展示。
- **是否必填**：导入校验是否要求非空；「否（有默认）」表示允许空，写入时由系统赋值。
- **中间表-下拉模糊搜索 / 仅手动输入**：与登记页控件对齐；两列一般为**一是一否**（PDF 定稿个别行为「是/是」的以 PDF 为准）。
- **字段校验**：能否沿用线上导入的字段校验及报错提醒，或统一为「字段名+不存在 / 格式错误」等（与 PDF 一致）。

**预填依据**：对照登记维修结算记录界面：带「请选择」、下拉三角的 → 下拉模糊=是；纯输入、日期、单选 → 仅手动=是。

## 五、业务规则汇总（与 PDF 一致）

1. **聚合**：仅 **维修编号** 相同 → 一张结算单；**维修编号必填**。  
2. **维修信息**：组内取 **第一行**；不一致不报错。  
3. **支付方**：不参与聚单判断；单据级展示值与 **第一行** 一致（空走默认）。  
4. **配件费**：空 → `单价×数量`；非空 → 导入值。  
5. **工时费、其他费**：独立按行导入；允许与配件费均为空（与现网一致）。  
6. **付款**：一行一条；允许付款方式重复。  
7. **备注**：逗号合并；截断至现网上限。  
8. **重复编号**：系统中已存在该维修编号 →在**提交落库（阶段二）** **拒绝**该单生成。  
9. **物料费**：用户不可通过导入编辑；逻辑同现网。  
10. **同编号原子提交**：同一 **维修编号** 分组内，**全部行**阶段一校验通过后，才允许随本次提交生成该单；**禁止**仅落库组内部分行。  
11. **中间表**：上传解析后数据进入中间态，**点击提交前**不写维修结算单主表/明细（与「导入物料申请」中间表语义一致）。  
12. **打开方式与手工录入**：点击**「导入」**展开中间表；用户可**仅通过手动增行**完成数据录入并提交，**不依赖**必须先上传 Excel。  
13. **上传覆盖**：每次成功解析上传文件后**覆盖**当前中间表内容；覆盖前若已有未提交数据须**确认**。

**说明**：第 3.2 条「导入成功」后字段不可改、保养侧增量字段等，见第三章与 **4.7**（与 PDF 表述一致，不单列编号避免与 PDF 冲突）。

## 七、验收要点（测试关注）

1. 点击**「导入」**展开中间表；**未点提交前**数据库（或业务上可观测的结算单列表）**无**对应新单。上传 Excel 或纯手填增行均适用。  
2. 中间表支持编辑后，**校验结果**列随阶段一规则更新；错误行修正后可通过阶段一。  
3. **同一维修编号** 多行中若 **1 行**阶段一仍失败，点击提交：**不得**为该编号生成结算单；**不得**只落库其余行；提示信息可理解。  
4. 同一维修编号 **全部行**阶段一通过后提交，生成 **一张**结算单，明细条数等于行数；无 5 条槽位限制。  
5. 阶段二：维修编号已存在于系统 → **拒绝**该单生成，且**不**产生缺明细的半单。  
6. 配件费为空且数量 3、单价 10 → 配件费 30；配件费填 25 → 以 25 为准。  
7. 三费全空行在中间表与落库后与现网录入一致（允许）。  
8. 两行同为现金付款 → 两条付款记录。  
9. 备注多行合并与截断符合现网上限。  
10. 费用合计与手工录入同数据结果一致（抽样对比）。  
11. 交互与「导入物料申请（车辆）」在步骤、提交、关闭提示上**无明显体验割裂**（文案可不同，流程一致）。  
12. **不上传文件**：仅手动添加多行、通过阶段一后提交，可正常生成结算单。  
13. **上传覆盖**：中间表已有数据时再次上传，解析结果**替换**表内全部行；**有确认提示**；覆盖后校验结果列与阶段一状态重置或重算（以实现为准）。
