### 需求背景

客户场景：维修厂提供的维修单多为竖版长明细。当前痛点如下：

1. 导入模板为**横版**布局，客户需复制粘贴适配模板后才能导入。  
2. 单次最多导入5 个项目，而客户单车一次维修常有20 余条明细。  
3. 本需求**直接升级**原导入能力（**同一业务入口**）：替换为**长表模板 + 中间表预览提交**，不再沿用「项目一～五」宽表与「仅事后结果文件」的旧交互；需配套更新下载模板与页面说明，减少对用户的认知断层。  
4. 维修/保养编号**可不填**：不填时由系统生成且**一行一单**；填写时支持**同编号多行聚合**为一单。

## 一、产品概述

### 1.1 背景与问题

财运通已支持维修结算单 Excel 导入，但现有模板采用「项目一～项目五」等**固定槽位**，明细条数受限；且模板要求用户维护合计类字段，与录入页「**先填项目明细、由系统自动计算费用合计**」的逻辑不一致，导致客户大单无法导入或重复做工。

### 1.2 产品定义

在**不改变「从 0 新建模块」**的前提下，对**现网维修结算单导入**做**直接替换式升级**：沿用用户已熟悉的**导入入口与菜单路径**，将能力升级为**长表**（一行一条项目明细）+ **中间表**（预览、校验、提交落库）。**聚单规则**：**维修编号为空**时，系统**自动生成**编号，**每行生成一张**结算单，**不聚合**；**维修编号非空**时采用用户维护的编号，**多行编号相同**则**聚合**为一张结算单。费用、默认值、付款行规则与线上一致并对齐本 PRD（保养侧编号规则同 **4.7**）。

### 1.3 产品目标

- 解除「每单固定 N 条项目」带来的导入上限问题；在**用户填写了维修/保养编号且多行同号**时，支持该编号下任意行数的项目明细合并为一单（上限若存在，遵循现网技术约束；本需求不另设业务封顶，除非现网已有）。**编号为空**时，**每行仅一条明细、一单**，无同批次聚合。  
- 导入模板与录入页一致：**费用合计由系统根据明细汇总计算**，用户无需在 Excel 中填写单据级合计列（若模板仍保留历史列，导入时**忽略**或后续从模板移除，见字段表说明）。  
- 降低分组歧义：**维修/保养编号非空**时，**相同编号**参与聚单；**编号、车牌号、维修类型**（保养侧为**保养编号、车牌号、保养类型**）在同一**聚合组**内须**完全相同**（**编号为空**时每组仅一行，无需跨行一致），该一致性在**点击提交时**做逻辑校验，失败则整体报错。**其余表头类非必填字段**可在多行中仅填首行，聚合落库取首行（见第四章「导入使用说明」与 5.2）。  
- 改善导入体验：由「仅事后报错文件」改为**中间表预览与行内修正**（参考现网「导入物料申请（车辆）」等能力），**提交前**可编辑；**正式落库**仅在用户点击「提交」后执行。

### 1.4 核心价值

| 维度 | 价值 |
|------|------|
| 客户侧 | 与客户纸质/对账单「一单多行明细」结构一致，大单可导入；减少在表格外手工汇总。 |
| 产品一致性 | 与录入页「明细驱动费用」一致，避免「表算一遍、系统再算一遍」。 |
| 实施 | 规则清晰（**空编号每行一单**、**同非空编号聚单**、首行表头、付款逐行），便于开发与测试。 |
| 可纠错性 | 中间表支持行内修改与校验结果展示，减少反复导出 Excel、核对报错行号的成本。 |
| 数据完整性 | **同一聚合组（同非空编号下的多行，或编号为空时的单行）原子提交**，避免部分落库导致单据缺明细或重复编号无法二次提交。 |

## 二、用户与场景

### 2.1 角色

| 角色 | 说明 |
|------|------|
| 车队财务 / 车务管理员 | 批量导入维修结算数据，核对生成单据。 |
| 系统管理员（可选） | 配置导入模板可见列（若现网支持模板配置）。 |

### 2.2 核心场景

1. 客户 Excel 中**同一非空维修编号**占多行，每行一条配件/工时类明细，导入后生成**一张**维修结算单，项目明细条数与行数一致；**维修编号留空**时，**每一行**生成**一张**结算单（每单一条明细，编号系统生成），**不与**其他空编号行聚合。  
2. 导入行中**支付方、付款方式**可为空，保存时按**现网默认值**赋值（与手工录入一致）。  
3. 每行对应**一条付款信息**；允许多行同为「现金」等，**不合并**为一条付款记录。  
4. **用户填写的**维修编号在系统中**已存在**时，在**提交落库阶段**校验并**拒绝**（不覆盖、不更新）。**系统生成的**编号须保证不与现存数据冲突（实现方式由研发保证）。  
5. 用户在中间表中对齐、修正数据后提交；**同一聚合组内所有行均校验通过后**，该单才允许生成，**禁止部分行先落库**。  
6. 用户点击**导入**展开中间表后，**可不传 Excel**，直接在表格中**手动新增行**并填写，与上传解析后的编辑、提交流程相同。

## 三、功能说明

### 3.1 功能清单

| 功能 | 描述 |
|------|------|
| 原导入能力升级 | **直接替换**现网维修/保养结算单的**原有导入**交互与模板：用户仍在**原入口**（列表页/登记流程中与现网一致的「导入」等）进入，但流程改为**长表模板 + 中间表**；旧版宽表模板**不再作为标准下载**（若用户手持旧文件上传，由研发定义明确报错或迁移提示，见附录「兼容」）。 |
| 中间表导入交互 | 用户在**上述原入口**点击**「导入」**后**展开**抽屉/弹窗，展示**可编辑中间表**；可在此**上传 Excel**将内容解析进表，也可**不上传**、仅靠**手动加行**维护数据。**提交前**不写入维修结算单主数据；形态参考现网「导入物料申请（车辆）」（模板下载、选文件、表格预览、校验结果列、提交、行首 +/- 增删行）。 |
| 两阶段校验 | **阶段一（仅前端校验）**：在中间表编辑过程中，**仅**做前端可完成的校验——如**必填项为空**、**字段格式不合法**（数字小数位、日期格式等）、**枚举/格式类输入非法**；结果展示在「校验结果」列（或行状态），用户可就地修改。**不**在此阶段做需服务端或跨行业务的逻辑判断（见下行「提交」）。**阶段一必填项（前端）**仅为「**维修类型/保养类型、车牌号**」两项非空及格式合法；**维修编号/保养编号非必填**（见第四章）。**提交时（逻辑校验 + 落库）**：用户点击「提交」后，由服务端执行**全部业务逻辑校验**（如：同一聚合组内编号/车牌/类型是否一致、**用户填写的**编号是否已在系统存在、组织/车辆/司机/字典等主数据是否存在、同组原子性等），**全部通过后**生成结算单；**失败则返回明确报错**（可定位行号/编号），**成功则返回成功提示**；不在中间表阶段做上述逻辑校验。 |
| 同组原子提交 | **先按聚单规则分组**（见「聚合规则」）：任一组内**只要存在一行未通过前端校验**，建议**不允许提交**或提交时由服务端拒绝该批（以实现为准）。**逻辑上**：提交时须保证同一聚合组**整单**通过校验后一次性生成，**不允许**仅落库组内部分行。**禁止部分提交**，避免生成「缺明细的单据」或再次提交时被判定**编号重复**而无法补全。 |
| 长表导入 | 一行一条项目明细；**维修/保养编号非空且多行相同**时合并为一张结算单；**编号为空**时每行单独一张结算单（编号系统生成）。 |
| 聚合规则 | **维修/保养编号为空**：**不聚合**，每行生成一张结算单，结算单号/维修（保养）编号由**系统自动生成**（规则与现网新建单据一致，须保证唯一）。**编号非空**：以用户填写为准；**多行编号相同**则聚合为**一张**结算单。**同一聚合组内车牌号、维修类型/保养类型须一致**在**提交时**由服务端校验（见「表头取值」）。 |
| 表头取值 | **维修编号/保养编号、车牌号、维修类型/保养类型**：在同一**聚合组**内，各行列值须**完全一致**（**编号为空**时每组仅一行，本条自动满足），在**点击提交时**做逻辑校验，不一致则**报错且不生成该单**。**其余表头类字段**（如司机、服务商等）：以**组内文件行序第一行**为准；允许**仅在首行填写**、后续行留空，多行均填写且与首行不一致时**以首行为准**（提交时是否提示由产品/研发定，本需求不强制前端拦截）。 |
| 配件费计算 | 配件费为空时按 **单价×数量** 写入；配件费非空则以导入值为准。 |
| 费用汇总 | 单据级配件费/工时费/其他费/费用合计由系统按明细汇总，与录入页逻辑一致。 |
| 三类费用校验 | **取消**「配件费、工时费、其他费不能同时为空」的导入专项校验；与**现网录入**一致，允许均为空。 |
| 支付默认值 | 支付方、付款方式**非必填**；为空时按现网默认（如公司付费、现金）赋值。 |
| 付款行 | **不聚合**；每行一条付款记录；允许重复付款方式。 |
| 备注合并 | **同一聚合组内**多行备注合并为一条，**英文逗号**分隔；超长按**现网该字段上限**截断（**编号为空**时每组一行，无多行合并）。 |
| 重复编号 | 与系统已有维修结算单**维修编号**冲突：**仅针对用户填写的编号**，**提交时** **拒绝**该编号对应整单的生成；**不在**中间表阶段做存在性校验（不得出现「半单已落库」）。系统生成的编号不应与库内已有编号冲突。 |
| 中间表数据带入 | 用户上传并解析 Excel 后，以**单元格内容原样**填入中间表（含空值、文本格式数字等）；**不在此阶段**执行业务聚合、费用重算、与系统主数据对账等**落库级逻辑**。用户可随时**手动新增/删除行**（与物料申请导入一致）。 |
| 上传覆盖中间表 | 每次上传并解析成功后，**覆盖当前中间表已有内容**（先清空再写入本次解析结果，或等价实现）；**不追加**到已有行后。用户手填未提交的数据被覆盖前需**二次确认**（文案与现网一致）。 |
| 中间表控件类型 | 各字段在中间表内**下拉模糊搜索**与**仅手动输入**的约定见第四章；本版已按**登记维修结算记录**页与录入组件一致性预填。 |
| 会话与关闭提示 | 未提交关闭弹窗/页面导致中间表数据丢失的，需有**与现网物料申请导入一致**的提示或行为（见参考交互）。 |

### 3.2 导入交互说明（中间表，参考「导入物料申请（车辆）」）

**目标**：解决原流程「只能拿到导入结果文件再改 Excel」的痛点；用户可在系统内预览、改错后再落库。

**整体步骤（与参考能力对齐）**

1. **下载模板**：支持「模板设置」「下载模板」（若现网维修导入已具备则沿用）。  
2. **打开中间表**：用户在**维修/保养结算单导入原入口**点击**「导入」**，**展开**抽屉/弹窗，展示中间表区域（可与物料申请导入一致：左侧步骤说明 + 右侧表格）。**此时即可手动点「+」增行填写**，无需先上传文件。  
3. **上传文件（可选）**：选择本地 Excel 后解析；文件大小、行数等限制**与现网一致或单独约定**。解析成功后，**覆盖当前中间表**（清空已有行再写入本次解析结果；若中间表已有未提交数据，覆盖前**确认提示**）。**上传解析成功后仍不落库**维修结算单。  
4. **表格内容**：列为本需求字段（见第四章）及**校验结果**列；支持行内编辑；各字段控件类型（模糊下拉 / 纯手输）见第四章标记。  
5. **阶段一（前端校验）**：对每行仅做**必填非空**、**格式/非法输入**等前端可判定项；不通过则在「校验结果」列展示原因（可多行错误合并）；用户可改单元格直至前端校验通过。**不**在此步请求服务端做业务逻辑校验。  
6. **提交（逻辑校验 + 生成）**：用户点击「提交」后，将当前中间表数据提交服务端，**一次性执行全部逻辑校验**（组内编号/车牌/类型一致、**用户填写编号**是否已存在、主数据是否存在、费用与原子性规则等）；**全部通过**则生成结算单并**返回成功**；**任一逻辑失败**则**返回报错**（含原因，可定位行/单），**不产生脏数据**（事务上整批回滚或仅失败单不落库，由研发实现，**不得**对同一**非空编号聚合组**只生成部分明细行）。  
7. **同聚合组原子提交（强制）**  
   - **分组规则**：**维修/保养编号非空**时，将行按**相同编号**分为一组；**编号为空**时，**每一行单独为一组**（每组成一单、一条明细）。  
   - 用户须先通过**前端校验**再点击提交；**提交后**由服务端判断该组是否满足全部逻辑规则。**生成成功后**，界面提示「**导入成功**」等终态，**字段值不允许修改**。  
   - **禁止**：逻辑校验未通过却部分落库；或**应聚合的多行**被拆成多单，导致**单据明细不完整**；或**用户填写编号已存在**却无法补全。

### 3.3 业务流程（落库阶段简述）

在用户于中间表点击**提交**，且**前端校验已通过**、**服务端逻辑校验与落库成功**后：

1. **分组**：**编号非空**时按**相同维修/保养编号**分组；**编号为空**时每行一组。组内行序以当前中间表行序为准（与用户调整后的顺序一致）。  
2. 每组生成一张维修结算单：**编号为空**时由系统生成单据编号/维修编号；**编号非空**则落库为用户填写值。表头字段取自该组**行序第一行**；项目明细为组内每行一条（编号为空时组内仅一行）；付款信息为组内每行一条（不合并）。  
3. 配件费按规则计算后，汇总费用合计；备注合并截断；支付默认值补全。  
4. 提交失败时服务端返回明确原因（行号、维修编号、失败类型）；**同一提交批次内**事务性要求见 3.2。

### 3.4 与旧模板差异（迁移说明）

| 项目 | 旧模板（现状） | 新模板（本需求后） |
|------|----------------|---------------------|
| 项目列 | 项目一～项目五等宽表槽位 | 长表：一行一条项目 |

## 四、字段全量说明（导入与业务）

以下表格为需求及约定。列含义如下：

- **是否支持导入模板配置**：是否允许在「导入模板配置」中控制该列是否出现在可下载模板中（若现网无配置能力，则全部为「是」，表示「应出现在标准模板列清单中」）。  
- **是否默认进入导入模板**：新建/重置模板时，该列是否默认勾选展示。  
- **是否必填**：**前端（阶段一）必填项仅为两项**：**维修类型/保养类型**、**车牌号**（见下表「是否必填」列）。**维修编号/保养编号**为**非必填**：空则提交后由系统生成且**该行（该行所在组）独立成单**；非空则参与聚单与重复性校验。下表中其余字段均为**非必填**。「否（有默认）」表示允许空，写入时由系统赋值。  
- **中间表-下拉模糊搜索**：在中间表内是否提供下拉 + 关键字模糊检索后选择（与登记页带「请选择」、下拉三角的控件对齐，以实现为准）。  
- **中间表-仅手动输入**：在中间表内是否以纯输入/日期选择/单选等为主（无上述检索下拉）。两列**一般为一是一否**；个别字段两列均为「是」的（如配件名称）仍保留。

**填写约定**：若业务需要「可搜可选又可手输」（ComboBox），可在「说明」列补充；若字段在中间表不展示或只读带出，两列填**不适用**。

## 五、业务规则汇总

1. **聚合**  
   - **维修/保养编号为空**：**不聚合**；**每行**生成**一张**结算单，编号由**系统自动生成**。  
   - **维修/保养编号非空**：**相同编号**的多行 → **一张**结算单；**前端必填**仅为 **维修类型/保养类型、车牌号**（两项）。  
2. **维修信息（表头）**  
   - **维修编号/保养编号、车牌号、维修类型/保养类型**在同一**聚合组**内须**逐行一致**，在**提交时**做逻辑校验，不一致则**报错且不生成**（**编号为空**时每组仅一行，本条自动满足）。  
   - **司机、服务商等其余非必填表头字段**：可**仅在首行录入**，后续行可空；聚合落库**默认取首行**。多行均填且不一致时**以第一行为准**。  
3. **支付方**：不参与聚单判断；单据级展示值与 **第一行** 一致（空走默认）。  
4. **配件费**：空 → 单价×数量；非空 → 导入值。  
5. **工时费、其他费**：独立按行导入；允许与配件费均为空（与现网一致）。  
6. **付款**：一行一条；允许付款方式重复。  
7. **备注**：**同一聚合组内**逗号合并；截断至现网上限（**编号为空**时每组一行，无多行合并）。  
8. **重复编号**：**用户填写的**维修/保养编号已在系统中存在 → **提交时** **拒绝**该单生成；系统生成编号须避免冲突。  
9. **物料费**：用户不可通过导入编辑；逻辑同现网。  
10. **同聚合组原子提交**：同一聚合组（**非空编号**下的多行，或**空编号**的**单行**）内，用户**前端校验通过**后可点击提交；**提交时**须整组逻辑校验通过才生成该单；**禁止**仅落库组内部分行。  
11. **校验分层**：**阶段一**仅**前端校验**（**车牌号、维修类型/保养类型**必填空值、格式/非法输入等）；**全部逻辑校验（含组内一致性、主数据、用户填写编号重复等）在提交时执行**，并**统一返回成功或报错**。  
12. **中间表**：上传解析后数据进入中间态，**点击提交前**不写维修结算单主表/明细（与「导入物料申请」中间表语义一致）。  
13. **打开方式与手工录入**：在**现网导入原入口**点击**「导入」**展开中间表；用户可**仅通过手动增行**完成数据录入并提交，**不依赖**必须先上传 Excel。  
14. **上传覆盖**：每次成功解析上传文件后**覆盖**当前中间表内容；覆盖前若已有未提交数据须**确认**。  
15. **上线策略**：本需求为**能力替换**而非并行双轨；**标准下载模板**与界面说明须同步切换为长表规则；若用户仍上传旧宽表文件，须有**明确提示**或迁移指引（见附录「兼容」）。

## 六、验收要点（测试关注）

1. **原导入入口**进入后即为**长表 + 中间表**流程（**非**改造前「仅上传即异步出结果」的旧交互）。点击**「导入」**展开中间表；**未点提交前**数据库（或业务上可观测的结算单列表）**无**对应新单。上传 Excel 或纯手填增行均适用。  
2. 中间表支持编辑后，**校验结果**列随**前端校验**规则更新；修正后可通过前端校验。  
3. **前端必填**仅为 **维修类型（或保养类型）、车牌号**；**维修/保养编号**可空；未补齐或格式非法时在前端提示；**主数据是否存在、用户填写编号是否重复等不在阶段一判定**。  
4. 同一聚合组内 **维修编号/保养编号、车牌号、维修类型/保养类型** 任一行与其它行不一致 → **提交时**服务端**报错**，不生成该单（**编号为空**的组仅一行，本场景不适用）。  
5. **同一非空编号**下多行中若 **1 行**仍**未通过前端校验**，应阻止提交或提交被服务端拒绝（以实现为准）；**不得**部分落库。  
6. **前端校验通过**后点击提交：**非空编号多行相同**时生成 **一张**结算单，明细条数等于该行数；**编号均为空**的多行应生成**多张**结算单（每行一单），每单**一条**明细；无 5 条槽位限制。  
7. **提交时**：**用户填写**的维修/保养编号已存在于系统 → **拒绝**该单生成，且**不**产生缺明细的半单；返回明确报错信息。  
8. 配件费为空且数量 3、单价 10 → 配件费 30；配件费填 25 → 以 25 为准。  
9. 三费全空行在中间表与落库后与现网录入一致（允许）。  
10. 两行同为现金付款 → 两条付款记录。  
11. **同一非空编号**下多行备注合并与截断符合现网上限；**编号为空**时每单仅一行备注，无合并。  
12. 费用合计与手工录入同数据结果一致（抽样对比）。  
13. 交互与「导入物料申请（车辆）」在步骤、提交、关闭提示上**无明显体验割裂**（文案可不同，流程一致）。  
14. **不上传文件**：仅手动添加多行、**前端校验通过**后提交，服务端校验通过可正常生成结算单。  
15. **上传覆盖**：中间表已有数据时再次上传，解析结果**替换**表内全部行；**有确认提示**；覆盖后**前端校验**状态重置或重算（以实现为准）。  
16. **导入使用说明**：模板/弹窗中含**红字**第 2～4 条（**车牌号+类型必填**、**编号选填及同号聚合规则**、**非空编号组内非必填可仅首行**）。  
17. **编号为空批量行**：多行**均不填**维修/保养编号时，提交后生成**多**张结算单，**不得**误合并为一单。  
18. **提交响应**：逻辑校验失败时返回**可理解的报错**；成功时返回**成功提示**（文案与现网一致或可配置）。
