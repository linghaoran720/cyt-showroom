**报销单和收据关联说明**：

**业务场景说明**：
- **报销单**：用于轮胎采购成本的报销，属于支出类财务单据
- **收据**：用于轮胎售卖收入的确认，属于收入类财务单据
- **两者独立**：一个轮胎可以同时有报销单和收据，互不影响

**报销单关联**：
- `reimbursement_no`：关联的报销单编号，有超链接，可以快速抽屉弹窗打开财务单据
- `reimbursement_status`：报销状态，用于记录轮胎的报销进度
  - 1：未报销 - 轮胎还未被纳入任何报销单
  - 2：部分报销 - 轮胎的采购金额被部分报销
  - 3：已报销 - 轮胎的采购金额已全部报销
- `reimbursement_amount`：已报销金额，用于记录轮胎已被报销的金额

**收据关联**：
- `receipt_no`：关联的收据编号，有超链接，可以快速抽屉弹窗打开财务单据
- `receipt_status`：收据状态，用于记录轮胎的收据审批进度
  - 1：待审批 - 轮胎的收据已创建，等待财务审批
  - 2：已审批 - 轮胎的收据已通过审批
  - 3：已拒绝 - 轮胎的收据已被拒绝

**典型业务场景**：
1. **场景1：采购新胎并报销**
   - 入库时填写采购价格 1280元
   - 创建报销单，报销采购成本
   - 报销单审批通过后，`reimbursement_status = 3`

2. **场景2：售卖旧胎并开收据**
   - 轮胎回收后，售卖给外部客户，售价 500元
   - 自动创建收据
   - 收据审批通过后，`receipt_status = 2`，轮胎状态变为"已卖出"

3. **场景3：采购、报销、售卖全流程**
   - 入库时采购价格 1280元 → 创建报销单 → 报销通过
   - 使用一段时间后回收 → 售卖 500元 → 创建收据 → 收据审批通过
   - 此时轮胎同时有报销单和收据记录

**关联信息显示**：
- 在轮胎详情页面中，显示关联的报销单编号和收据编号
- 编号显示为超链接，点击可以快速抽屉弹窗打开财务单据
- 样式参考创建报销单和创建收据的样式
- 信息不可编辑，仅用于查看和追踪

### 2.4 数据字典

**轮胎状态（current_status）及状态流转**：

```
入库（1）──→ 已领用（2）──→ 已回收（3）──→ 已作废（4）──→ 售卖待确认（5）──→ 已卖出（6）
  │            ✗ 不可卖出        │                  │                  ▲
  └────────────────────────────┴──────────────────┴──────────────────┘
                              可卖出（需先回收）
```

**状态流转说明**：
- 入库 → 领用 → 回收 → 作废 → 售卖待确认 → 已卖出（标准流程）
- 入库 → 卖出（直接售卖新胎）
- 已回收 → 卖出（售卖回收的旧胎）
- 已作废 → 卖出（售卖报废轮胎）
- **已领用状态不可直接卖出**，必须先回收

| 状态 | 值 | 允许的操作 | 不允许的操作 |
|------|---|------------|------------|
| 入库 | 1 | 领用、卖出 | 作废、回收 |
| 已领用 | 2 | 回收 | 领用、卖出、作废 |
| 已回收 | 3 | 领用、卖出、作废 | - |
| 已作废 | 4 | 卖出 | 领用、回收 |
| 售卖待确认 | 5 | 无（等待收据审批） | 所有操作 |
| 已卖出 | 6 | 无（终态） | 所有操作 |

**事件类型（event_type）**：
- 1：入库
- 2：领用
- 3：回收
- 4：作废
- 5：卖出
- ~~6：维修记录~~（已移除，维修不记录事件）

**轮胎类型（tire_type）**：
- 1：新胎
- 2：旧胎

**使用范围（use_scope）**：
- 1：本级及下级（该轮胎可在入库网点及其下级网点使用）
- 2：本级（该轮胎仅可在入库网点使用）
- 3：全集团（该轮胎可在集团内所有网点使用）

**安装位置（install_position）**：
- 前左
- 前右
- 后左1
- 后右1
- 后左2
- 后右2
- 备胎

## 三、核心业务逻辑设计

### 3.1 入库单号生成规则

**规则**：网点名称 + 年后2位 + 月2位 + 日2位 + 流水号(4位)

**示例**：
- 总部仓库260330001（网点名称：总部仓库，日期：2026-03-30，流水号：0001）
- 总部仓库260330002
- 北京分公司260330001


### 3.2 轮胎入库（手动操作）

**触发时机**：用户在轮胎管理模块手动添加轮胎

**前置条件**：
- 轮胎编号必填（客户自己录入，唯一）
- 轮胎品牌必填
- 轮胎尺寸必填
- 轮胎类型必填（新胎/旧胎）
- 采购价格必填
- 采购日期必填
- 使用范围必填（默认"本级及下级"）

**业务逻辑**：
1. 校验轮胎编号唯一性
2. 生成入库单号
3. 创建轮胎卡片，状态设置为"入库"
4. 记录使用范围信息
5. 记录入库事件


```

### 3.3 批量新增轮胎功能

**功能描述**：用户在输入框中粘贴多个轮胎编号，系统自动创建多行轮胎卡片

**支持的分隔符**：
- 逗号（,）
- 空格（ ）
- 换行符（\n）
- 制表符（\t）

**输入示例**：
```
123,456,321,234
或
123 456 321 234
或
123
456
321
234
```

**业务逻辑**：
1. 解析输入框中的编号，按分隔符分割
2. 去除空白字符和重复编号
3. 校验每个编号的唯一性
4. 批量创建轮胎卡片
5. 返回成功/失败的统计信息


```

### 3.4 领用流程业务逻辑

**前置条件**：
- 轮胎状态必须为"入库"或"已回收"
- 车辆必须存在
- 安装位置不能为空

**业务逻辑**：
1. 校验轮胎状态
2. 校验车辆是否存在
3. 更新轮胎卡片：
   - 状态改为"已领用"
   - 记录当前车辆
   - 记录安装位置
4. 记录领用事件

### 3.5 回收流程业务逻辑

**前置条件**：
- 轮胎状态必须为"已领用"

**业务逻辑**：
1. 校验轮胎状态
2. 更新轮胎卡片：
   - 状态改为"已回收"
   - 清空当前车辆
   - 清空安装位置
3. 记录回收事件

### 3.6 作废流程业务逻辑

**前置条件**：
- 轮胎状态必须为"已回收"

**业务逻辑**：
1. 校验轮胎状态
2. 更新轮胎卡片：
   - 状态改为"已作废"
3. 记录作废事件

### 3.7 卖出流程业务逻辑

**前置条件**：
- 轮胎状态必须为"入库"、"已回收"或"已作废"
- **不允许"已领用"状态的轮胎直接卖出**（需先回收）

**业务逻辑**：
1. 校验轮胎状态（必须为"入库"、"已回收"或"已作废"）
2. 更新轮胎卡片：
   - 状态改为"售卖待确认"
3. 自动创建收据
4. 记录卖出事件

### 3.8 维修记录（手动关联）

**前置条件**：
- 轮胎可以在任何状态下记录维修
- 维修单据ID可选（可以不关联维修单据，仅记录维修金额）
- 维修成本必填
- 同一维修单据不可多次关联同一轮胎（防止重复记录）
- 同一维修单据可以关联不同的轮胎

**业务逻辑**：
1. 校验维修单据关联：
   - 如果填写了维修单据ID，校验该单据是否已关联过该轮胎
   - 如果已关联，返回错误提示："该维修单据已关联过此轮胎"
2. 更新轮胎卡片：
   - 累计维修次数 +1
   - 累计维修成本 += 本次维修成本
3. 记录维修单据关联关系（如果填写了维修单据ID）

**说明**：
- 不创建维修事件记录
- 仅更新轮胎卡片的维修次数和累计成本
- 维修单据关联关系用于防止重复关联，不影响其他业务逻辑

## 四、功能流程与UI交互设计

### 4.1 轮胎入库流程

#### 4.1.1 流程概述

**流程概述**：
用户点击"轮胎入库"按钮 → 打开轮胎入库抽屉弹窗 → 填写基本信息 → 添加轮胎明细 → 提交

**流程图**：
```
点击"轮胎入库"按钮
        ↓
打开轮胎入库抽屉弹窗
        ↓
填写基本信息（入库来源、联系人、入库时间、采购方等）
        ↓
添加轮胎明细
  ├─ 单个添加
  └─ 批量录入编号
        ↓
填写轮胎信息（品牌、尺寸、规格、型号、单价等）
        ↓
提交入库
        ↓
生成入库单号和轮胎卡片
```

#### 4.1.2 批量录入对话框设计

**触发方式**：在轮胎入库抽屉弹窗中，点击"批量录入轮胎编号"按钮

**对话框内容**：
- 标题：批量录入轮胎编号
- 提示文本：批量录入支持的分隔符：逗号、空格、换行
- 输入框：多行文本框，用于粘贴轮胎编号
- 按钮：提交、取消

**交互逻辑**：
1. 用户在输入框中粘贴轮胎编号（支持多种分隔符）
2. 点击"提交"按钮
3. 系统解析编号，识别出编号数量
4. 关闭对话框，返回轮胎入库抽屉
5. 根据识别出的编号数量，自动增加轮胎明细表格的行数
6. 每行的轮胎编号字段自动填充对应的编号

**示例**：
- 输入：123,456,321,234
- 识别数量：4
- 自动生成4行，编号分别为：123、456、321、234

#### 4.1.3 轮胎入库抽屉弹窗设计

**弹窗结构**：

##### 4.1.3.1 基本信息部分（上部）

| 联系人 | 文本 | 是 | 入库操作的联系人 |

**表格列说明**：
- 序号、轮胎编号、当前状态：显示信息，不可编辑，字段置灰
- 品牌、尺寸、规格、型号、单价、轮胎类型：根据轮胎状态决定是否可编辑
  - 可编辑的字段：正常显示，用户可输入
  - 不可编辑的字段：置灰显示，用户无法点击或输入

**修改规则**：

修改权限基于轮胎的报销状态和采购价格：

**1. 轮胎状态为"入库"时**：
- 所有字段都可修改（品牌、尺寸、规格、型号、单价、轮胎类型）
- 这些字段正常显示，用户可以编辑

**2. 轮胎状态为"售卖待确认"时**：
- 所有字段都不可修改
- 所有字段置灰显示，用户无法编辑
- 原因：已创建财务收据，不允许修改任何信息

**3. 轮胎已报销（reimbursement_status = 3）时**：
- 所有字段都不可修改
- 所有字段置灰显示，用户无法编辑
- 原因：已报销的轮胎不允许修改任何信息，以保证财务数据的准确性

**4. 轮胎未报销或部分报销（reimbursement_status = 1 或 2）时**：
- 如果采购价格为空或为0（回填场景）：
  - 仅可修改采购价格
  - 采购价格：正常显示，用户可以编辑
  - 其他字段：置灰显示，用户无法编辑
- 如果采购价格不为空且不为0：
  - 所有字段都不可修改
  - 所有字段置灰显示，用户无法编辑
  - 原因：已有其他操作（领用、回收、作废、卖出）的轮胎不允许修改，除非需要回填价格

**修改限制**：
- 轮胎编号：始终置灰，不可修改（轮胎编号是唯一标识）
- 入库单号：始终置灰，不可修改（必须按照入库单进行修改，不可跨入库单修改）
- 当前状态：始终置灰，不可修改（轮胎的当前状态由系统自动管理）

**UI设计细节**：
- 不可编辑的字段使用灰色背景或灰色文字表示
- 不可编辑的字段禁用鼠标点击和键盘输入
- 不可编辑的字段显示禁用光标（如：not-allowed）
- 可编辑的字段使用白色或浅色背景，支持正常的输入交互

**底部按钮**：
- 取消：关闭抽屉弹窗，不保存任何修改
- 确认修改：保存修改

#### 4.1.6 入库单修改的校验规则

**前端校验**：
1. **必填字段校验**：
   - 基本信息部分的必填字段不能为空
   - 轮胎明细部分的必填字段不能为空

2. **轮胎编号唯一性校验**：
   - 修改后的轮胎编号在该入库单内不能重复
   - 修改后的轮胎编号在整个系统内不能重复

3. **修改权限校验**：
   - 根据轮胎的报销状态和采购价格，校验该字段是否可修改
   - 如果轮胎状态为"入库"：所有字段可修改
   - 如果轮胎状态为"售卖待确认"：所有字段不可修改
   - 如果轮胎已报销（reimbursement_status = 3）：所有字段不可修改
   - 如果轮胎未报销或部分报销（reimbursement_status = 1 或 2）：
     - 如果采购价格为空或为0：仅可修改采购价格
     - 如果采购价格不为空且不为0：所有字段不可修改
   - 如果尝试修改不可修改的字段，显示错误提示

4. **采购价格校验**：
   - 如果轮胎状态为"售卖待确认"：不允许修改采购价格
   - 如果轮胎已报销（reimbursement_status = 3）：不允许修改采购价格
   - 如果轮胎未报销或部分报销（reimbursement_status = 1 或 2）且采购价格为空或为0：允许修改采购价格
   - 如果轮胎未报销或部分报销（reimbursement_status = 1 或 2）且采购价格不为空且不为0：不允许修改采购价格

**后端校验**：
1. **轮胎编号唯一性校验**：
   - 校验修改后的轮胎编号是否在系统内重复

2. **轮胎状态校验**：
   - 校验轮胎的当前状态是否与前端显示一致
   - 防止并发修改导致的状态不一致

3. **入库单一致性校验**：
   - 校验所有修改的轮胎是否都属于同一个入库单号
   - 防止跨入库单修改

4. **数据一致性校验**：
   - 校验修改后的数据是否符合业务规则

   - 如果被勾选，则不算冲突，不显示提醒

2. **位置不冲突**：

   - 校验轮胎状态是否为"入库"或"已回收"
   - 如果不是，返回错误

4. **使用范围校验**：
   - 校验轮胎的使用范围是否包含当前用户所在网点
   - 如果不包含，返回错误

### 4.6 创建报销单操作设计

#### 4.6.1 流程概述

**创建报销单操作**是指根据轮胎入库单创建财务报销单据的过程。该操作支持：
- 针对单个入库单的轮胎创建报销单（勾选任一条轮胎或全部轮胎）
- 支持批量勾选多个入库单的轮胎进行报销单创建
- 记录报销状态：已报销、未报销、部分报销
- 记录已报销金额和未报销金额
- 部分报销时，报销金额按照采购单价均摊到轮胎，采用4舍5入规则
- 与财务系统集成，创建财务报销单据

**流程入口**：轮胎列表页面 → 选择轮胎 → 点击"创建报销单"按钮

**流程输出**：
- 生成财务报销单据
- 更新轮胎的报销状态
- 记录报销金额和分摊信息

**报销金额分摊示例**：
- 入库单包含：A轮胎采购价100元，B轮胎采购价50元，总计150元
- 报销总金额：30元
- 分摊计算：
  - A轮胎占比：100/150 = 66.67%，分摊金额 = 30 × 66.67% = 20.00元
  - B轮胎占比：50/150 = 33.33%，分摊金额 = 30 × 33.33% = 10.00元
  - 验证：20.00 + 10.00 = 30.00元 ✓

#### 4.6.3 报销金额分摊逻辑

**分摊规则**：
1. **按采购单价比例分摊**：
   - 分摊金额 = 报销总金额 × (该轮胎采购价格 / 所有轮胎采购价格之和)

2. **4舍5入规则**：
   - 每条轮胎的分摊金额保留两位小数
   - 采用标准的4舍5入规则

3. **余数处理**：
   - 由于4舍5入，可能出现分摊金额之和与报销总金额不相等的情况
   - 将余数分配给分摊金额最大的轮胎
   - 例如：报销30元，A轮胎分摊20.00元，B轮胎分摊10.00元，余数0.00元

**分摊示例**：
- 入库单包含：A轮胎采购价100元，B轮胎采购价50元，C轮胎采购价50元，总计200元
- 报销总金额：33元
- 分摊计算：
  - A轮胎：33 × (100/200) = 16.50元
  - B轮胎：33 × (50/200) = 8.25元
  - C轮胎：33 × (50/200) = 8.25元
  - 验证：16.50 + 8.25 + 8.25 = 33.00元 ✓

### 4.8 取消操作设计

#### 4.8.1 流程概述

**取消操作**是指撤销已执行的轮胎操作，将轮胎状态恢复到操作前的状态。该设计支持：
- 删除入库（轮胎初始状态，可直接删除）
- 取消领用（将轮胎从"已领用"状态变回"入库"或"已回收"）
- 取消回收（将轮胎从"已回收"状态变回"已领用"）
- 取消报废（将轮胎从"已作废"状态变回"已回收"）
- 取消售卖（将轮胎从"售卖待确认"状态变回售卖前的状态）
- 支持批量取消
- 需要填写取消原因
- 记录取消事件和操作日志
- 权限控制（仅特定角色可取消）

**取消操作的状态转换**：

| 原操作 | 原状态 | 取消操作 | 目标状态 | 说明 |
|-------|-------|---------|---------|------|
| 轮胎入库 | 入库 | 删除入库 | 已删除 | 删除入库记录，轮胎卡片标记为已删除 |
| 轮胎领用 | 已领用 | 取消领用 | 入库/已回收 | 轮胎返回库存，清除车辆关联信息 |
| 轮胎回收 | 已回收 | 取消回收 | 已领用 | 轮胎重新分配给原车辆 |
| 轮胎报废 | 已作废 | 取消报废 | 已回收 | 轮胎恢复为可用状态 |
| 轮胎售卖 | 售卖待确认 | 取消售卖 | 入库/已回收/已作废 | 轮胎重新纳入库存，恢复到售卖前的状态 |

**操作入口**：
- 轮胎列表页面 → 选择轮胎 → 点击"取消操作"按钮 → 选择要取消的操作类型（下拉菜单）

- 报销单编号、收据单号、维修单号等详细信息可在轮胎详情中查看

#### 4.10.3.1 操作列设计

**操作列特点**：
- 第一列固定显示，不随表格滚动而隐藏
- 仅对单条轮胎数据有效
- 根据轮胎状态动态显示可执行的操作

**操作按钮显示规则**：

| 轮胎状态 | 显示的操作按钮 |
|---------|--------------|
| 入库 | 领用、详情 |
| 已领用 | 回收、详情 |
| 已回收 | 领用、报废、售卖、详情 |
| 已作废 | 售卖、详情 |
| 售卖待确认 | 详情（已创建财务收据，不可进行其他操作） |
| 已卖出 | 详情 |

**操作按钮说明**：
- 领用：将轮胎从"入库"或"已回收"状态变为"已领用"
- 回收：将轮胎从"已领用"状态变为"已回收"
- 报废：将轮胎从"已回收"状态变为"已作废"
- 售卖：将轮胎从"入库"、"已回收"或"已作废"状态变为"售卖待确认"
- 详情：打开轮胎详情抽屉弹窗，显示轮胎的详细信息和事件日志

**交互方式**：
- 单条操作：点击操作按钮，直接执行该操作（与勾选一条轮胎点击对应的操作按钮一致）
- 不需要勾选，直接点击操作按钮即可执行
- 操作完成后，表格自动刷新，显示最新的轮胎状态和可执行的操作

#### 4.10.4 搜索和过滤功能

**默认筛选项**（财运通表格引擎支持配置）：
- 入库单号：文本搜索
- 轮胎编号：文本搜索
- 车牌号：文本搜索
- 轮胎状态：下拉选择（入库、已领用、已回收、已作废、售卖待确认、已卖出）
- 领用人：文本搜索
- 入库时间：日期范围选择（默认近1个月）

**搜索和过滤说明**：
- 财运通表格引擎支持配置字段显示和筛选项显示
- 用户可以根据需要添加或移除筛选项
- 支持多条件组合搜索（AND逻辑）
- 搜索结果实时显示

#### 4.10.5 操作按钮

**页面级操作按钮**（轮胎列表页面顶部）：
- 轮胎入库：新增轮胎
- 轮胎领用：批量发放轮胎给车辆（需要勾选轮胎）
- 轮胎回收：批量回收轮胎（需要勾选轮胎）
- 轮胎报废：批量标记轮胎为报废（需要勾选轮胎）
- 轮胎售卖：批量售卖轮胎给外部客户，自动创建收据（需要勾选轮胎）
- 创建报销单：根据入库单创建财务报销单据（需要勾选轮胎）
- 维修记录：关联维修单据并记录维修成本（需要勾选一条轮胎）
- 取消操作：撤销各个操作（需要勾选轮胎）

**操作列按钮**（轮胎列表第一列，单条操作）：
- 根据轮胎状态显示可执行的操作
- 仅对单条轮胎数据有效
- 不需要勾选，直接点击操作按钮即可执行
- 详情：始终显示，点击打开轮胎详情抽屉弹窗

**批量操作限制**：
- 所有批量操作默认200条上限
- 如果选择超过200条，系统会提示"最多只能选择200条"

## 五、集成方案设计（MVP简化版）

### 5.1 与车辆管理模块集成

**集成方式**：API调用（仅查询）

**轮胎模块调用车辆模块**：
- 查询车辆信息
- 校验车辆是否存在

### 5.2 与维修保养模块集成（手动关联）

**集成方式**：手动关联维修单据

**维修流程**：
1. 用户在轮胎管理模块中，手动输入维修单据ID
2. 输入本次维修成本（≤维修单据成本）
3. 系统记录维修事件，更新轮胎卡片的维修次数和累计成本

**不涉及自动集成**：
- 不监听维修保养模块的事件
- 不调用维修保养模块的API
- 维修单据ID仅用于记录和追溯，不做强制校验

## 九、权限控制


**功能权限**：
- 轮胎卡片查看
- 轮胎卡片编辑
- 轮胎领用
- 轮胎回收
- 轮胎维修
- 轮胎报废
- 统计分析查看

**数据权限**：
- 按网点隔离（入库网点）
- 按集团隔离
- 按使用范围隔离：
  - 本级及下级：该轮胎可被入库网点及其下级网点查看和使用
  - 本级：该轮胎仅可被入库网点查看和使用
  - 全集团：该轮胎可被集团内所有网点查看和使用

**操作权限限制**：
- 领用操作：仅当前用户所在网点在轮胎的使用范围内时，才能进行领用操作
- 回收操作：仅当前用户所在网点在轮胎的使用范围内时，才能进行回收操作
- 编辑操作：仅入库网点的用户可以编辑轮胎信息
- 删除操作：仅入库网点的用户可以删除轮胎

**权限管理配置**：
- 财运通有权限管理模块，轮胎管理模块的每个功能都可以单独配置权限
- 权限可以合成，比如"轮胎入库/入库修改/入库撤销"可以合成一个权限"入库管理权限"
- 其他操作类似，可以合成权限组：
  - 入库管理权限：轮胎入库、入库修改、入库撤销
  - 领用管理权限：轮胎领用、撤销领用
  - 回收管理权限：轮胎回收、撤销回收
  - 报废管理权限：轮胎报废、撤销报废
  - 售卖管理权限：轮胎售卖、撤销售卖
  - 财务管理权限：创建报销单、收据审批
  - 维修管理权限：维修记录
  - 统计分析权限：统计分析查看
- 权限配置由系统管理员在财运通权限管理模块中进行设置


## 十一、MVP范围说明

### 11.1 MVP阶段的设计原则

**独立性**：轮胎管理模块在MVP阶段完全独立运作，不依赖物料模块和仓库管理模块。

**简洁性**：
- 轮胎入库改为手动操作
- 维修单据手动关联，不做强制校验
- 移除仓库管理的概念
- 简化数据模型

**可扩展性**：
- 数据库设计预留扩展空间
- 业务逻辑清晰，便于后续集成
- 事件表设计支持后续的事件驱动架构

### 11.2 后续集成计划（非MVP）

**第二阶段**：与物料模块集成
- 自动生成轮胎卡片
- 自动出入库

**第三阶段**：与仓库管理模块集成
- 支持多仓库管理
- 仓库间转移

**第四阶段**：与维修保养模块深度集成
- 自动关联维修单据
- 维修流程自动化
